## 流水线契约 — 本技能边界声明

> **本技能只负责：情节蓝图 + 上下文衔接 + 章节结构。**
>
> **本技能的工作模式**：支持两种建纲模式——**滚动建纲（默认）**：先建前10章细纲+衔接包链，进入03写作；每写完5章滚动补齐后续5-10章。**全书建纲（可选，≤30章时适用）**：一次性产出全书逐章细纲。两种模式均使用分批策略防止AI偷懒合并章节，产出的每章细纲自包含衔接包，供03逐章扩写。
>
> **本技能不负责**：
> - ❌ 质检规则（大纲五检/质量Gate/衔接包自检/详细度检查/衰减检测）→ 这些属于 **02b_细纲质检.md**
> - ❌ 正文质感规则（AI味检测/感官倒置/脏感注入/物理代价）→ 这些属于 **04精修师**
> - ❌ 长期记忆与状态压缩（记忆锚点存储/状态压缩/伏笔生命周期管理）
> - ❌ 能力剥离与阶段检测（依赖期判定/辅助隐藏/成长评估）
> - ❌ 映射表变异/安全违规
>
> **本技能的交付物**：每章细纲 + 衔接包 + 记忆锚点标注 + 章级情绪蓝图（V4.5新增）+ 伏笔追踪表（V4.5新增）。这些是02b质检和03扩写系统的唯一输入。
>
> **衔接包是02a的唯一生产物**：03扩写系统只能读取衔接包，无权修改。03自行校验衔接包禁止矛盾，并在扩写后自检。
>
> **数据所有权声明（单一数据源原则）**：
> - **衔接包的唯一生产者是02a**。02b质检、03扩写系统只能读取衔接包，无权修改。03逐章取用衔接包，并在扩写后自检禁止矛盾。
> - **记忆锚点（P0/P1/P2）的协作管理**：P0锚点初始值来自00的JSON元数据。02a在每章产出衔接包时同步更新P0锚点快照（角色位置/伤势/持有物/关系/伏笔状态）。02b在每5章回顾时校验锚点一致性，发现漂移在下一章衔接包中修正。
> - 衔接包是"施工图纸"，03扩写时必须严格遵守禁止矛盾清单。03在自检阶段如发现正文与禁止矛盾冲突 → 修正正文，不改衔接包。
> - **衔接包**（Part G）字段分为P0必须和P1可选两级（V5.2.0新增）：
>   - P0必须字段（限300字以内）：角色状态、禁止矛盾、数值锁定、角色知识边界
>   - P1可选字段（不强制）：前章回顾、必须延续细节、状态锁定、闪回标注、本章新增状态、质感预算、跨章去重清单
> - **记忆锚点**（Part I.6）必须标注 P0/P1/P2 级别，P0 级锚点在每章细纲中强制附带。
> - **JSON 元数据**（由设定架构师产出）中的 anti_patterns.forbidden/required 列表须由用户在调用本技能时粘贴到上下文中，本技能 YAML 头已声明引用 00 设定文件但实际加载依赖用户操作。
>
> **模式感知**：可在粘贴前添加 `/mode_full` `/mode_assisted` `/mode_hardcore` 指令控制辅助强度。

### 模式使用指南

粘贴本文件为系统Prompt前，根据当前模式选择：
- `/mode_full`：粘贴全文，所有规则和建议全开
- `/mode_assisted`（默认）：粘贴全文，AI静默运行，减少主动建议
- `/mode_hardcore`：**仅粘贴 Part G（衔接包）+ Part F2（章节结构）+ Part 6（输出格式）**，删除爽感保护/流氓逻辑/黄金三章等主动干预模块

# Part N: 爽感保护协议（番茄核心·题材适配）

## N.0 题材判断（V5.2.0新增）

读取 `novel_meta.genre` 字段，按题材启用不同的开篇抓手和爽点要求：

| 题材 | 开篇抓手 | 爽点要求 |
|------|---------|---------|
| 玄幻/都市/系统 | 爽点（前500字必须有） | 每3章≥1个小爽点，每10章≥1个大爽点 |
| 悬疑/推理 | 悬念（前500字必须有1个未解之谜） | 信息揭示即爽点，不强制传统爽点 |
| 言情/情感 | 情感碰撞（前500字必须有1次关系张力） | 关系推进/反转即爽点 |
| 严肃文学 | 氛围锚点（前500字必须有1个沉浸感细节） | 不强制爽点，以情绪到达为标准 |
| 历史/军事 | 利益冲突（前500字必须有1次博弈） | 权谋胜利即爽点 |

> **题材混合时**：双题材混合取两题材要求的并集（更严格者优先）；主题材权重≥70%时以主题材要求为主。

## N.1 为什么这是番茄的核心？

番茄的编辑通常在**前3章（甚至前1章）**就决定生死。

**开篇没有爽点 = 必死**（注：此条针对玄幻/都市/系统题材；悬疑/言情等题材以各自开篇抓手为准）

爽点类型（以玄幻/都市/系统为主）：
- 打脸（主角被看不起 → 反转 → 对方傻眼）
- 金手指觉醒（获得外挂/系统/传承）
- 绝境翻盘（必死之局 → 意外破局）
- 复仇成功（仇人出现 → 主角出手 → 爽）
- 装逼（主角展现隐藏实力 → 众人震惊）

## N.2 爽感保护协议

```
[爽感保护协议 - N模块]

第1章强制检查：
├─ 是否有爽点事件？（玄幻/都市/系统题材）
│   ├─ 是 → 通过检查
│   └─ 否 → ⚠️ 警告："第1章无爽点，番茄签约风险极高！"
│       └─ 建议：
│           ├─ A. 将后文章节的爽点挪到第1章
│           ├─ B. 在第1章结尾加入"爽点预告"（主角获得某种暗示）
│           └─ C. 重新设计开篇，直接从爽点场景切入
│
├─ 爽点是否在前500字内出现？
│   ├─ 是 → ✅ 完美开篇
│   ├─ 否，但在前1000字内 → ⚠️ 可接受，但建议前移
│   └─ 否，超过1000字 → ❌ 警告："铺垫太长，读者已流失"
│
└─ 爽点是否有"情绪着陆"？
    ├─ 是 → ✅ 爽感完整
    └─ 否 → ⚠️ 建议："打完脸后，让主角有个'没意思'的反应，显得更酷"

前三章检查（黄金三章）：
├─ 第1章：必须有爽点（打脸/金手指/翻盘）— 题材适配后以各自开篇抓手为准
├─ 第2章：必须有新钩子（让读者想知道接下来会发生什么）
├─ 第3章：必须有更大的期待感（升级的爽感预告）
│
└─ 任一章节不满足 → ⚠️ 警告："黄金三章未达标，签约风险高"
```

## N.3 爽感熔断器

```
[爽感熔断器]

目的：防止爽点写太长，变成水字数

规则：
├─ 单个爽点描写不超过800字
│   └─ 超过 → ⚠️ 警告："爽点过长，建议精简"
│
├─ 爽点必须有"情绪着陆点"
│   └─ 示例：打完脸后，主角觉得没意思，然后走人
│   └─ 这样显得主角很酷，不care
│
├─ 爽点后必须有"新钩子"
│   └─ 示例：打脸后，主角发现更大的麻烦来了
│   └─ 让读者期待下一章
│
└─ 爽点不能连续出现
    └─ 连续爽点会麻木
    └─ 建议：爽点 → 小挫折 → 更大的爽点
```

### 爽点类型库
- **打脸**：被看不起→反转→对方傻眼。主角全程不解释，用行动说话
- **金手指觉醒**：绝境/意外→获得外挂→实力暴涨。觉醒过程要快，觉醒后立刻展示效果
- **绝境翻盘**：必死之局→意外破局→反杀。绝境要真绝，翻盘要合理（有伏笔）
- **复仇成功**：仇人出现→主角出手→爽。仇人要先嚣张，主角出手要狠，反应要淡
- **装逼**：主角展现隐藏实力→众人震惊。先让众人低估，主角不经意展露，反应要夸张

## N.5 爽感密度调节器

```
[爽感密度调节器]

章节类型与爽感密度：
├─ 第1章：超高密度（必须有爽点）
├─ 第2-3章：高密度（必须有钩子）
├─ 常规章节：中密度（每章至少一个小爽点）
├─ 过渡章节：低密度（可以平淡，但要有期待感）
└─ 高潮章节：超高密度（连续爽点，但要控制节奏）

爽感间隔规则：
├─ 连续爽点不超过3个
│   └─ 超过 → 爽感麻木
│   └─ 建议：爽点 → 小挫折 → 爽点
│
├─ 无爽点章节不超过2章
│   └─ 超过 → ⚠️ 警告："连续无爽点，读者可能流失"
│
└─ 大爽点间隔不超过5章

不强制传统爽感密度规则。

# Part J: 动机显隐周期表（行动派主角）

## J.1 为什么动机要显隐？

番茄读者不喜欢：
- 主角想太多（内耗）
- 主角动机太复杂（看不懂）
- 主角太纠结（不爽）

番茄读者喜欢：
- 主角目标明确（知道要干嘛）
- 主角行动果断（说干就干）
- 主角动机简单（为钱/为仇/为女人）

## J.2 动机显隐周期表

```
[动机显隐周期表 - J模块]

显性期（前5章）：
├─ 主角目标必须极其简单粗暴
├─ 错误示范："我想探寻世界的真理，但又怕受伤，还要考虑师父的感受..."
├─ 正确示范："为了赏金，老子要弄死这个怪物。"
│
├─ 表面动机：钱、仇、女人、生存、面子
├─ 行动动机：弄死他、抢过来、逃出去、打他的脸
└─ 深层动机：隐藏起来，不要让读者觉得主角在装

隐性期（5章后）：
├─ 可以逐渐揭示深层动机
├─ 但不要长篇大论，用行动暗示
└─ 示例：主角为了钱杀人 → 但杀完不要钱 → 读者自己悟：原来他是为了...

周期规则：
├─ 第1-3章：纯表面动机（钱/仇/生存）
├─ 第4-5章：表面动机 + 行动动机
├─ 第6-10章：开始暗示深层动机
├─ 第10章后：深层动机逐渐浮现
└─ 注意：深层动机永远不要直接说出来，让读者自己悟
```

### 动机类型速查
| 类型 | 表面动机 | 行动 | 深层动机 |
|------|---------|------|---------|
| 利己型 | 为了钱/资源 | 抢/杀/争 | 证明自己/填补空虚 |
| 复仇型 | 为了报仇 | 弄死仇人 | 为了正义/守护 |
| 生存型 | 为了活下去 | 逃/打/拼 | 寻找意义 |
| 守护型 | 为了保护某人 | 挡/杀/争 | 弥补过去的遗憾 |
| 证明型 | 为了证明自己 | 挑战/超越/打脸 | 获得认可 |

```
[行动派主角塑造]

原则：想干就干，干了再说

错误示范（想太多）：

# Part F2: 章节字数与结构控制（关键）

## F.1 问题：固定结构导致同质化

**原有问题**：固定"引入-推进-高潮-收束"四段式，导致每章结构雷同，像流水线产品。

**正确理念**：每章结构应由**剧情需要**决定，而非套固定模板。

```
❌ 错误思维：每章都要有"引入-推进-高潮-收束"
 结果：每章都像同一个模子刻出来的

✅ 正确思维：这章剧情需要什么结构，就用什么结构
 结果：每章都有自己的形态，跟随剧情自然流动
```

## F.2 剧情驱动型结构

### F.2.1 章节结构类型库

不同剧情需要不同的结构，以下是常见结构类型：

```
[章节结构类型库]

类型A：连续推进型
├─ 适用：追逐、逃亡、连续战斗、紧凑剧情
├─ 结构：场景1 → 场景2 → 场景3 → 场景4（无明确高潮点，全程推进）
├─ 特点：节奏快，场景间紧密衔接，不强调单点高潮
└─ 示例：主角一路逃亡，每场景都是逃亡的一部分

类型B：单点爆发型
├─ 适用：对峙、揭秘、决战、重大转折
├─ 结构：铺垫(30%) → 爆发(50%) → 余波(20%)
├─ 特点：有明确高潮点，前后都为高潮服务
└─ 示例：两人对峙，最后真相揭露

类型C：日常流动型
├─ 适用：日常、关系发展、氛围营造
├─ 结构：场景1 → 场景2 → 场景3（无明显高潮，情绪自然流动）
├─ 特点：节奏慢，重在氛围和细节，不追求戏剧性
└─ 示例：两人相处一天，关系微妙变化

类型D：信息揭示型
├─ 适用：调查、探索、发现线索
├─ 结构：发现1 → 分析 → 发现2 → 串联 → 新方向
├─ 特点：信息量递增，最后形成新认知
└─ 示例：主角调查真相，逐步拼凑线索

类型E：情绪沉淀型
├─ 适用：事件后的消化、内心转变、决策
├─ 结构：回忆/反思 → 情绪处理 → 做出决定
├─ 特点：内敛，重在心理，动作少
└─ 示例：主角经历大事件后，独自消化，最终做出选择

类型F：多线并进型
├─ 适用：多视角、多地点、同时发生的事件
├─ 结构：线A场景1 → 线B场景1 → 线A场景2 → 线B场景2
├─ 特点：视角切换，制造张力
└─ 示例：主角和配角同时在不同地点行动

类型G：倒叙/插叙型
├─ 适用：揭示过去、补充背景、制造悬念
├─ 结构：当下 → 回忆 → 当下（带着新理解）
├─ 特点：时间跳跃，信息差
└─ 示例：当下触发回忆，揭示角色过去

类型H：起承转合型
├─ 适用：完整的小故事弧、独立事件
├─ 结构：起(25%) → 承(30%) → 转(30%) → 合(15%)
├─ 特点：有完整弧光，但不是每章都必须
└─ 示例：一次完整的对话/冲突/解决
```

### F.2.2 如何选择章节结构

```
[章节结构选择指南]

问自己三个问题：

问题1：这章的剧情目标是什么？
├─ 推进主线？ → 类型A/B/F
├─ 揭示信息？ → 类型D/G
├─ 发展关系？ → 类型C
├─ 情绪沉淀？ → 类型E
└─ 完整事件？ → 类型H

问题2：这章的节奏应该快还是慢？
├─ 快 → 类型A/B/F
├─ 慢 → 类型C/E
└─ 中 → 类型D/H/G

问题3：这章在整卷中的位置？
├─ 卷初 → 需要抓人，倾向B/A
├─ 卷中 → 可以多样，根据剧情需要
└─ 卷末 → 需要收束或爆发，倾向B/H

问题4：01传递的情感温度是什么？（来自01分卷大纲的"本卷情感温度图谱"）
├─ 🔥 高温（紧张/愤怒/恐惧）→ 倾向 类型B（单点爆发）/ 类型F（多线并进）
├─ 🌡️ 中温（好奇/期待/担忧）→ 倾向 类型A（连续推进）/ 类型D（信息揭示）
├─ ❄️ 低温（平静/疲惫/日常）→ 倾向 类型C（日常流动）/ 类型E（情绪沉淀）
└─ 如果01未传递情感温度（使用旧版大纲）→ 根据问题1-3自行判断

禁止：
├─ ❌ 不问剧情需要，机械套用"引入-推进-高潮-收束"
├─ ❌ 每章都用同一种结构
└─ ❌ 为了凑结构而强行加场景
```

## F.3 字数控制原则

```
[字数控制 - 目标4000字]

基本原则：
├─ 每章目标字数：4000字（±500字，即3500-4500字）
├─ 字数根据场景重要性分配，不是平均分配
├─ 每个场景标注预估字数，但结构由剧情决定
└─ 场景数：3-5个，根据剧情需要决定

场景字数参考（按重要性）：
├─ 核心场景：1200-1800字（最重要，详写）
├─ 重要场景：800-1200字（次重要，正常写）
├─ 过渡场景：400-800字（连接用，简写）
└─ 点缀场景：200-400字（氛围/细节，点到为止）

总字数保证：
├─ 核心场景 × 1 + 重要场景 × 2 = 约4000字
├─ 或：重要场景 × 3 + 过渡场景 × 1 = 约4000字
└─ 或：核心场景 × 1 + 重要场景 × 1 + 过渡场景 × 2 = 约4000字
```

### F.4.5 镜头焦距指令（解决AI抓不住重点）

> **核心问题**：AI缺乏"镜头感"和"信息权重意识"。如果细纲里写"两人走进客栈"，AI会觉得"走进客栈"和"两人拔剑生死相搏"具有同等的信息权重，从而花300字描写客栈的桌椅板凳。
>
> **解决方案**：在细纲的场景规划中，强制标注每个场景的"镜头焦距"，让AI明确知道这个场景该详写还是略写。

```
[镜头焦距指令 - 每个场景必须标注]

镜头类型：

[特写镜头：放慢时间，感官放大]
├─ 适用：情感爆发点、关键动作、重大转折、角色对峙、爽点事件
├─ 要求：
│   ├─ 放慢叙事节奏，放大感官细节
│   ├─ 每个动作拆解为2-3步描写
│   ├─ 允许环境描写承载情绪
│   └─ 允许角色内心活动（控制在合理比例）
├─ 字数倾向：该场景占本章总字数的30-50%
└─ 示例标注：[特写镜头：放慢时间，感官放大]

[过场镜头：一笔带过，禁止环境铺陈]
├─ 适用：赶路、进门、日常过渡、时间流逝、地点切换
├─ 硬性约束：
│   ├─ ❌ 禁止描写环境建筑（不要写客栈的桌椅板凳、街道的店铺招牌）
│   ├─ ❌ 禁止描写天气（除非天气变化直接推动剧情）
│   ├─ ❌ 禁止描写无意义的路人（不要写路人甲的表情、动作、对话）
│   ├─ ❌ 禁止超过200字（过场就是过场，不是正文）
│   └─ ✅ 直接切入核心动作（从"进门"直接跳到"坐下说话"）
├─ 字数倾向：该场景占本章总字数的5-10%，最多200字
└─ 示例标注：[过场镜头：一笔带过，禁止环境铺陈]

[中景镜头：正常叙事节奏]
├─ 适用：一般性对话、信息交代、日常互动、铺垫段落
├─ 要求：正常节奏，不刻意加快也不放慢
├─ 字数倾向：该场景占本章总字数的15-25%
└─ 示例标注：[中景镜头：正常叙事]

镜头分配规则：
├─ 每章特写镜头：1-2个（核心场景必须是特写）
├─ 每章过场镜头：≥1个（过渡场景强制过场镜头）
├─ 每章中景镜头：剩余场景
├─ 禁止：连续3个场景都是特写（读者会疲劳）
└─ 禁止：连续2个场景都是过场（读者会觉得内容空洞）

过场镜头硬性约束（违反即重写）：
├─ 如果场景标注为[过场镜头]，则扩写时必须遵守：
│   ├─ 该场景总字数 ≤ 200字
│   ├─ 不得出现环境描写超过1句
│   ├─ 不得出现"只见/但见/放眼望去"等环境引入句式
│   └─ 必须在3句话内完成场景切换，切入下一场景的核心动作
└─ 常见错误示例：
    ❌ "两人走进客栈，只见大堂里摆着八仙桌，桌上放着茶壶，墙上挂着字画，角落里坐着几个客人..."（过场镜头禁止环境铺陈）
    ✅ "两人走进客栈，要了间雅间。茶上来的时候，张三开口了。"
```

```
[字数不足诊断]

├─ 症状：只有1-2个场景，写完才2000字
├─ 对策：根据剧情需要拆分场景
└─ 示例：

    拆分后：[场景1] 对峙 → [场景2] 交手 → [场景3] 反杀 → [场景4] 余波

原因2：场景描写太薄
├─ 症状：每个场景只写动作，没有感官/心理/环境
├─ 对策：每个场景至少包含2-3层描写
└─ 3层描写：
    ├─ 第1层：动作链（谁做了什么）
    ├─ 第2层：感官+心理（看到什么、感受到什么）

├─ 症状：对话2-3轮就结束，没有拉扯感
├─ 对策：重要对话至少5-8轮，加入动作间隔
└─ 注意：对话长度要符合角色性格和场景氛围

原因4：没有"呼吸空间"
├─ 症状：全程紧凑，没有环境描写和情绪沉淀
├─ 对策：在关键动作之间插入"呼吸片段"
└─ 呼吸片段：
    ├─ 角色独处时的感官体验（100-200字）
    ├─ 环境描写承载情绪（100-200字）
    └─ 日常碎片（倒水、看窗外、走路）（50-100字）
```

## F.8 正文章节内容拆分规则

### F.8.1 什么时候需要拆分章节？

```
[正文章节拆分触发条件]

需要拆分的情况：
├─ 单章正文超过6000字 → 强制考虑拆分
├─ 单章包含多个完整事件 → 按事件拆分为多章
├─ 单章时间跨度太大（如跨越多天）→ 按时间节点拆分
├─ 单章地点变换太多 → 按地点拆分
└─ 单章情绪弧太多（多个高潮点）→ 按情绪弧拆分

不需要拆分的情况：
├─ 单章正文在3500-4500字 → 正常，不拆
├─ 内容是连续动作（战斗、追逐、逃亡）→ 不拆，保持连贯性
├─ 拆分会破坏情绪完整性 → 不拆
└─ 拆分后某章太短（<2500字）→ 不拆，保持原样
```

### F.8.2 拆分原则：自然不突兀

```
[拆分原则]

核心原则：
├─ 拆分点必须是"自然的断点"，不是"硬切"
├─ 拆分后每章都能独立成立，有完整的情绪弧
├─ 拆分点要有"钩子"，让读者想继续读下一章
└─ 每章字数控制在3500-4500字

自然断点的类型：
├─ 时间断点：一天结束、夜晚入睡、清晨醒来、季节变换
│   └─ 示例："这一夜，他没睡好。" → 下一章从清晨开始
│
├─ 地点断点：离开某地、到达某地、进入新空间
│   └─ 示例："他转身离开了仓库。" → 下一章从路上开始
│
├─ 事件断点：一件事结束、新事件即将开始
│   └─ 示例："这件事算是告一段落了。" → 下一章从新事件开始
│
├─ 情绪断点：情绪转换、沉默、决定之后
│   └─ 示例："他沉默了很久，然后做出决定。" → 下一章从执行决定开始
│
└─ 信息断点：获得关键信息后的消化、悬念留白
    └─ 示例："真相已经浮出水面，但还有太多疑问。" → 下一章从追问开始

禁止的拆分方式：
├─ ❌ 在动作中间硬切（如"他举起刀——"然后切到下一章）
├─ ❌ 在对话中间硬切（除非是故意制造悬念）
├─ ❌ 在情绪高潮中间硬切
├─ ❌ 在战斗/追逐中间硬切
└─ ❌ 没有任何过渡，直接跳到新场景
```

### 拆分核心原则
- 拆分点选自然断点（时间/地点/事件/情绪/信息断点）
- 禁止在动作/对话/情绪高潮中间硬切
- 拆分后每章独立成立，3500-4500字

### F.8.4 拆分后的章节衔接

```
[章节衔接处理]

前章结尾的处理：
├─ 留钩子：悬念、危机、决定、新信息、异常
├─ 不要完全结束：留一点"未完成"的感觉
└─ 示例：
    ├─ 悬念式："但他不知道，更大的危机还在后面。"
    ├─ 危机式："路上似乎有人跟踪。"
    ├─ 决定式："他做出了决定——明天就出发。"
    └─ 异常式："一切都很顺利，顺利得有些不对劲。"

后章开头的处理：
├─ 快速衔接：不要大段回顾"上一章发生了什么"
├─ 直接切入：从动作/场景/情绪开始
└─ 示例：
    ├─ 承接动作："他没回头，但手已经摸到了刀柄。"
    ├─ 承接时间："第二天清晨，他出发了。"
    ├─ 承接情绪："那一夜，他没睡好。"
    └─ 承接场景："路上，雨越下越大。"

禁止的衔接方式：
├─ ❌ 后章开头大段回顾："上一章说到李四得知了真相..."
├─ ❌ 前章结尾完全结束，后章重新开始
├─ ❌ 前后章情绪断裂（前章紧张，后章突然日常）
└─ ❌ 拆分点前后没有呼应
```

> 拆分核心原则同上。每章独立，3500-4500字，拆分后细纲同步更新。

### F.8.6 拆分检查清单

```
[拆分检查清单]

拆分前检查：
├─ 字数是否真的需要拆？（>6000字才考虑）
├─ 内容是否包含多个可独立的事件？
├─ 是否存在自然断点？
└─ 拆分后每章字数是否合理？（3500-4500字）

拆分后检查：
├─ 拆分点是否自然？（读者会不会觉得突兀）
├─ 每章是否独立成立？（各有完整情绪弧）
├─ 章节衔接是否流畅？（前尾后头是否呼应）
├─ 钩子是否存在？（读者是否有动力继续）
├─ 字数是否合理？（每章3500-4500字）
└─ 细纲是否同步更新？
```

# Part G: 上下文衔接保障（关键）

## G.1 问题：扩写时前后不搭

**现象**：用细纲扩写正文时，AI不会翻看前文内容，导致：
- 前文角色受了伤，本章活蹦乱跳
- 前文角色在A城，本章突然出现在B城
- 前文刚建立的设定，本章完全遗忘
- 前文角色之间的关系，本章表现不一致

**根本原因**：AI扩写时只有当前章的细纲，没有前文的上下文。

## G.2 解决方案：细纲自带"衔接包"（V5.2.0 P0/P1分级）

```
[衔接包机制 — V5.2.0 P0/P1字段分级]

核心思路：
├─ 每章细纲必须自带"前文衔接信息"
├─ 扩写者不需要翻看前文，只看细纲就能保证连贯
└─ 衔接包是细纲的一部分，不是额外文件

衔接包字段分为两级（V5.2.0新增）：

P0必须字段（限300字以内 — 缺失=02b质检不通过）：

# Part H: 细纲定位修正（关键）

## H.1 问题：细纲变成了对话脚本

**现象**：生成的细纲像"对话脚本"，而非"情节蓝图"。

**原因**：
1. 文档对"核心场景"的定位不够清晰——AI误以为细纲就是写对话
2. v3.0的"对话潜台词"模块可能被过度强调，导致AI把重心放在对话上
3. 缺少"细纲 vs 正文"的明确区分——细纲应该描述"发生什么"，正文才写具体对话

## H.2 细纲的本质定位

```
[细纲定位 - 必须明确]

细纲是什么：
├─ 细纲是"蓝图"，不是"正文"
├─ 细纲描述"发生什么"，不写出具体对话
├─ 细纲标注"情绪/氛围/关键动作"，不写完整场景
└─ 细纲的读者是"扩写者"（可能是AI或人类）

细纲不是什么：
├─ ❌ 细纲不是对话脚本（不写出完整对话）
├─ ❌ 细纲不是正文缩写（不写完整场景描写）
├─ ❌ 细纲不是大纲（比大纲更具体，但不如正文详细）
└─ ❌ 细纲不是填空题（不是机械填充模板）
```

## H.3 细纲 vs 大纲 vs 正文

| 维度 | 大纲 | 细纲 | 正文 |
|------|------|------|------|
| 详细度 | 一句话概括 | 场景级描述 | 完整描写 |
| 对话 | 无 | 关键台词提示 | 完整对话 |
| 场景 | 无 | 场景概括+情绪 | 完整场景 |
| 字数 | 50字/章 | 500-800字/章 | 2000-5000字/章 |
| 读者 | 编辑/作者 | 扩写者 | 终端读者 |

## H.4 核心场景的正确写法

```
[核心场景正确写法]

必须包含：
├─ 场景概括：一句话描述"发生什么"
├─ 涉及角色：列出参与角色
├─ 情绪基调：标注场景的情绪
├─ 关键动作/转折点：列出关键动作
├─ 关键台词提示：给出1-2句台词的提示（不是完整对话）
└─ 后续影响：说明这个场景的后果

禁止写法：
├─ ❌ 写出完整对话（这是正文的工作）
├─ ❌ 写出完整场景描写（这是正文的工作）
├─ ❌ 只写一句话（这是大纲，不是细纲）
└─ ❌ 机械填充模板（要有实际内容）
```

### 细纲正反例
- ❌ **错误（对话脚本式）**：写出完整对话 + 无场景概括 + 无情绪标注 + 无关键动作
- ✅ **正确（蓝图式）**：场景概括（1句）+ 情绪基调 + 关键动作列表 + 台词提示（非完整对话）+ 后续影响

## H.6 细纲生成约束Prompt

```
[细纲生成约束 - 必须遵守]

细纲是"蓝图"，不是"正文"。请遵循以下规则：

1. 核心场景：
   - 用"场景概括"描述发生什么（一句话）
   - 标注"情绪基调"
   - 列出"关键动作/转折点"
   - 给出"关键台词提示"（不是完整对话，是提示）
   
2. 禁止：
   - ❌ 写出完整对话（最多给出1-2句台词提示）
   - ❌ 写出完整场景描写
   - ❌ 把细纲写成对话脚本
   - ❌ 只写一句话（那是大纲）

3. 字数要求：
   - 每章细纲：500-800字（不含元数据），黄金3章可达1000字
   - 每个核心场景：50-100字
   
4. 示例：
   ✅ "李四质问张三，张三最终说出真相，李四震惊"

# Part I: 详细度一致性保障（生成侧）

## I.1 问题：前面详细，后面一句话带过

**现象**：前几章细纲很详细，中后章越来越简略，最后变成"一句话带过"。

**更严重的现象**：中后期的细纲变成"几章合在一起介绍"——一个故事弧只用一段话概括，而不是每章单独介绍。

```
❌ 错误示例（多章合并介绍）：

Ch.15-18：李四前往C城寻找父亲的线索。
他在路上遇到了张三，两人一起调查。
最终发现父亲当年是被人陷害的。
李四决定复仇。

问题：
├─ 4章内容只用一段话概括
├─ 每章的具体场景、情绪、转折全部缺失
├─ 扩写者根本不知道每章该写什么
└─ 这不是细纲，这是大纲

✅ 正确示例（逐章独立介绍）：

Ch.15：李四独自出发
├─ 核心场景数：3
├─ 情绪心电图：压抑 → 紧张 → 决然
├─ 感官锚点：清晨的冷风、背包的重量
├─ 关键动作：告别 → 上路 → 第一夜露营
└─ 字数分配：4000字

Ch.16：路上遇张三
├─ 核心场景数：3
├─ 情绪心电图：意外 → 警惕 → 勉强同行
├─ 感官锚点：路边的篝火、张三身上的酒味
├─ 关键动作：发现跟踪 → 对峙 → 达成临时合作
└─ 字数分配：4000字

Ch.17：调查线索
├─ 核心场景数：4
├─ 情绪心电图：困惑 → 发现 → 震惊 → 疑虑
├─ 感官锚点：旧档案的霉味、照片上模糊的人影
├─ 关键动作：找到旧档案 → 发现矛盾 → 追问知情人 → 得到新线索
└─ 字数分配：4000字

Ch.18：真相浮出
├─ 核心场景数：4
├─ 情绪心电图：紧张 → 震惊 → 愤怒 → 决心
├─ 感官锚点：雨夜、闪电照亮的面孔
├─ 关键动作：找到关键证人 → 听到真相 → 确认陷害 → 决定复仇
└─ 字数分配：4000字
```

**原因**：
1. **AI注意力衰减**：长文本生成时，AI对后面章节的注意力下降
2. **上下文窗口限制**：生成到后面时，前面的约束条件被"遗忘"
3. **缺少详细度强制检查**：没有机制确保每章详细度一致
4. **一次性生成过多**：试图一次生成太多章节，导致质量递减
5. **AI偷懒合并**：AI倾向于把相似章节合并概括，减少输出量

## I.2 最低详细度要求（硬性约束）

```
[最低详细度要求 - 每章必须满足]

⚠️ 最高优先级约束：逐章独立，禁止合并
├─ 每一章必须有独立的完整细纲
├─ 禁止将多章合并为一段介绍（如"Ch.15-18：..."）
├─ 禁止用一个故事弧概括多章内容
├─ 每章细纲必须包含完整的：核心场景、情绪心电图、感官锚点、字数分配
└─ 即使是过渡章节，也必须有独立的3个以上核心场景

核心场景：
├─ 核心场景数 ≥ 3
└─ 每个场景必须有：场景概括 + 情绪基调 + 关键动作

情绪心电图：
├─ 必须标注
└─ 包含：开头 → 中段 → 高潮 → 收束

感官锚点：
├─ 至少1个
└─ 包含：核心锚点 + 围绕展开

字数要求：
├─ 每章细纲：500-800字（不含元数据）
├─ 每个核心场景：100-200字
└─ 确保细纲信息足够详细，扩写者能直接使用

检查机制：
├─ 每章生成后自动检查
├─ 任一项不合格 → 强制补充，不允许跳过
└─ 检查结果写入自检报告
```

### 分批策略
- 每批最多5章。每批生成后强制检查详细度
- 第N批详细度 < 第1批 × 80% → 触发警告
- 第N批详细度 < 第1批 × 50% → 严重警告，建议重写

> **注**：详细度检查协议、详细度衰减检测、详细度评分等质检规则已剥离至 02b_细纲质检.md。

# Part I.5: 每卷黄金3章设定（关键）

## I.5.1 什么是"黄金3章"？

**定义**：每卷的前3章是该卷的"黄金3章"，决定读者是否继续阅读本卷。

**作用**：
1. 确立本卷的基调/风格/核心冲突
2. 重新抓住读者的注意力（上一卷结束可能已疲劳）
3. 铺设本卷的核心线索和人物关系

## I.5.2 黄金3章的设定要求

```
[黄金3章设定 - 每卷必须]

每卷开始前，必须先设定本卷的"黄金3章"：

═══════════════════════════════════════
【卷X 黄金3章设定】
═══════════════════════════════════════

[本卷基调]
├─ 整体基调：热血/压抑/悬疑/温馨/悲壮...
├─ 节奏特点：快节奏推进/慢热铺垫/张弛有度...
├─ 情绪走向：从X到Y（如：从压抑到爆发）
└─ 核心冲突：本卷要解决的核心矛盾是什么？

│   ├─ 本卷核心冲突的第一次暗示
│   └─ 与上卷结尾的衔接（如有）
├─ 禁止：
│   ├─ ❌ 缓慢的开场（大段环境描写）
│   ├─ ❌ 与本卷主线无关的内容
│   └─ ❌ 让读者困惑的信息过载
└─ 示例：直接从危机场景切入，或从异常事件开始

[黄金第2章] 确立基调
├─ 目标：让读者清楚本卷"要讲什么"
├─ 必须包含：
│   ├─ 本卷核心冲突的正式呈现
│   ├─ 主角的目标/动机/困境的明确
│   └─ 本卷关键角色的登场或铺垫

│   ├─ ❌ 仍在游离，不进入主线

│   ├─ 读者对主角的情感投入（担心/期待）
│   └─ 本卷风格的稳定确立
├─ 禁止：
│   ├─ ❌ 仍是铺垫，没有推进
│   └─ ❌ 风格摇摆不定
└─ 示例：第一个障碍出现，主角做出选择，后果显现
```

## I.5.3 黄金3章与普通章节的区别

| 维度 | 黄金3章 | 普通章节 |
|------|---------|---------|
| 详细度 | 最高（细纲600-1000字） | 正常（细纲500-800字） |
| 钩子强度 | 必须有强钩子 | 可以是弱钩子 |
| 信息密度 | 高（快速确立基调） | 正常 |

## 6.2.1 衔接包归档与压缩机制（V5.0新增 — 防止衔接包链膨胀）

> ⚠️ **痛点对症**：到第50章时衔接包链有6-7万字，AI无法全部加载。
> 本机制将超过10章的旧衔接包自动压缩为摘要，将上下文负担从6万字降至约5000字。

**归档触发条件**：当衔接包链长度超过10章时，最早的非最近10章衔接包自动进入归档流程。

**归档流程**：
1. 提取核心信息，压缩为3-5行摘要：
   ```
   Ch.X | [一句话核心事件] | 引入：[关键角色/物品/伏笔] |
   [主角状态：枚数/境界/位置/伤势] | 悬念：[本章遗留钩子]
   ```
2. 原始衔接包移至 `细纲/衔接包链/归档/Ch.X_衔接包.md`
3. 摘要写入 `细纲/衔接包链/历史摘要.md`

**归档摘要示例**：
```
Ch.3 | 阿介首次猎杀荒兽，获2枚血核 | 引入：老长老、血核概念 |
阿介：2枚/凡骨/左肩轻伤 | 悬念：母亲去哪了
Ch.4 | 阿介被选入猎队，与沙九首次冲突 | 引入：沙九、猎队编制 |
阿介：2枚/凡骨/左肩痊愈 | 悬念：沙九为何针对阿介
```

**按需回溯机制**：当后续章节需要参考某个旧衔接包的详细信息时，AI可按章节号查找 `归档/Ch.X_衔接包.md`。02a在生成新衔接包时，如发现当前章需要回溯旧章细节，在衔接包中标注 `[需回溯Ch.X]`，提示用户粘贴该归档包。

**卷末批量归档**：每完成一卷时，执行批量归档（见卷末归档流程）。

## 6.2.2 卷末归档流程（V5.0新增 — 长篇中后期工程管理）

> ⚠️ **痛点对症**：到第100章时伏笔追踪表50+条、角色状态表30+角色、上下文管理模板极长。
> 本流程在每卷结束时执行归档，控制上下文规模。

**触发时机**：每完成一卷后执行（通常30-50章/卷）。

**归档操作清单**：

1. **衔接包归档**：
   - 本卷所有衔接包移至 `归档/第X卷/`
   - 生成卷级摘要（1段话概括本卷核心事件链）
   - 保留最近10章的完整衔接包不归档

2. **伏笔追踪表清理**：
   - CLOSED状态的伏笔移至 `追踪/伏笔归档.md`，标注回收章号
   - ABANDONED状态的伏笔标注废弃原因后归档
   - ACTIVE状态的伏笔保留，检查是否超过预警期（埋设超过30章未回收→标记⚠️）
   - 归档后追踪表只保留当前卷的活跃伏笔

3. **角色状态表精简**：
   - 当前卷未出场的角色移至"非活跃角色"区（只保留名字+最后出场章号+关键状态）
   - 已死亡角色移至"已退场角色"区（保留死亡信息供回溯）
   - 当前卷活跃角色保留完整状态

4. **上下文管理模板压缩**：
   - 已完成卷的进度信息压缩为1行摘要：`第X卷(Ch.X-Ch.Y)：[核心事件] | [角色变化] | [伏笔进展]`
   - 保留最近1卷的详细状态
   - 快速恢复命令更新为最新卷的最新章

5. **项目日志摘要**：
   - 每卷结束时生成"卷摘要"（核心事件/角色变化/伏笔进展/读者反馈如有）
   - 旧的项目日志条目压缩为摘要
