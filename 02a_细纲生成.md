## 在完整流水线中的位置

> **当前技能**：第 3a 步 — **细纲生成**（中观桥梁 — "施工图纸"生成模块）
> **核心使命**：将01产出的分卷大纲拆解为逐章细纲（含章节蓝图五段式+字数预算+多线情节+人物关系出场）。每章细纲自包含——03扩写者无需翻看前文即可开始写作。
> **上游**：← 00设定架构师（JSON元数据/角色档案/关系矩阵）← 01大纲构建师（分卷大纲/情绪曲线）
> **下游**：→ 02b质检（独立审查）→ 03a扩写（消费细纲+衔接包+感官锚点）
> **建纲策略**：滚动建纲（默认，先10章每5章补5-10章）/ 全书建纲（/mode_full，≤30章）
> **衔接包是02a唯一生产物**，03只能读取无权修改。03自检发现禁止矛盾→修正正文，不改衔接包。
> **番茄特供原则**：爽感第一，逻辑第二，文学性第三。前三章定生死，第1章定留存。

## 衔接包P0字段速查（首因效应·置于开头）

> **最高优先级**：P0字段是02a的核心交付物，缺失=02b质检不通过。P0字段限300字以内。

| P0字段 | 内容 | 信息来源 |
|--------|------|---------|
| **[角色当前状态]** | 关键角色的物理+心理状态+战力等级+生死状态 | 物理状态表+心理日记锚点+00的power_system/number_anchors |
| **[禁止矛盾]** | 本章绝对不能出现的内容（伤势/位置/关系/设定矛盾） | 物理状态表+设定文件 |
| **[数值锁定]** | 本章涉及的关键数值（年龄/距离/枚数/境界/物品数量/时间线/生死状态） | 00的number_anchors字段 |
| **[角色知识边界]** | 在场角色"已知"和"未知"的关键信息+信息差标注 | 前章衔接包的[本章新增状态] |

> P0生成时同步更新锚点快照：P0-5物理状态/P0-6战力等级/P0-7关键数值（每章必更新）。
> 已死亡角色必须标注"已死亡(Ch.X)"，本章如涉及须为闪回/回忆。

# Part F0: 伏笔主动健康检查（V5.4新增 — 借鉴InkOS伏笔主动检索）

> **核心问题**：长篇写到50章后，埋在20章前的伏笔容易被遗忘。当前伏笔追踪表是被动的——需要人工/LLM主动查看才决定是否推进。
> **解决方案**：每章细纲生成前，自动扫描伏笔追踪表中的"超期伏笔"，注入到章节意图的 `must_address` 字段中。

## F0.1 伏笔健康检查流程

```
伏笔健康检查(每章细纲生成前执行):
├─ 1. 读取伏笔追踪表（00设定JSON中的foreshadowing_tracker或独立文件）
├─ 2. 扫描所有 status=ADVANCE 或 status=PLANT 的伏笔
├─ 3. 计算每条伏笔的"超期程度":
│   ├─ last_advanced_chapter = 上次推进的章节号
│   ├─ current_chapter = 当前要写的章节号
│   ├─ gap = current_chapter - last_advanced_chapter
│   └─ 超期阈值: gap > 10章 → 超期 / gap > 20章 → 严重超期
├─ 4. 生成伏笔健康报告:
│   ├─ 超期伏笔列表（按gap降序）
│   ├─ 严重超期伏笔（gap>20）→ 必须在本章或下2章内推进
│   └─ 待回收伏笔（status=HINT，接近预期回收节点）→ 建议本章回收
├─ 5. 注入到章节意图:
│   ├─ must_address: [严重超期伏笔ID列表] → 本章细纲必须包含推进
│   ├─ should_address: [超期伏笔ID列表] → 本章细纲建议包含推进
│   └─ may_reap: [待回收伏笔ID列表] → 本章细纲可考虑回收
└─ 6. 02b质检时检查:
    ├─ must_address中的伏笔是否在细纲中推进？→ 未推进=质检不通过
    └─ should_address中的伏笔是否在细纲中推进？→ 未推进=警告（不阻断）
```

## F0.2 伏笔健康度评分

每条伏笔的健康度 = f(gap, importance, expected_reap_chapter)

| 健康度 | 判定 | 处理 |
|--------|------|------|
| 🟢 健康 | gap ≤ 10章 | 正常追踪 |
| 🟡 超期 | 10 < gap ≤ 20章 | should_address注入 |
| 🔴 严重超期 | gap > 20章 | must_address注入 |
| ⚫ 过期 | gap > 30章且无推进计划 | 标记为"遗失伏笔"，建议在卷末归档时清理或补推进 |

## F0.3 伏笔健康检查输出格式

```yaml
伏笔健康报告 — 第X章:
  总伏笔数: N
  健康: [FS-001, FS-003, ...]
  超期(should_address): 
    - FS-002: 最后推进Ch.15, gap=12, 摘要: [伏笔内容]
  严重超期(must_address):
    - FS-005: 最后推进Ch.8, gap=22, 摘要: [伏笔内容]
  待回收(may_reap):
    - FS-007: status=HINT, 预期回收Ch.30, 当前Ch.28
  过期(遗失):
    - FS-009: 最后推进Ch.3, gap=35, 建议: [补推进或归档清理]
```

## F0.4 与现有伏笔追踪表的集成

> 伏笔追踪表5状态生命周期不变（PLANT→ADVANCE→HINT→REAP→ARCHIVE），本检查是在生命周期基础上的"主动性增强"。

- 伏笔追踪表由02a维护（现有机制不变）
- 健康检查读取追踪表数据，不修改追踪表
- 健康检查结果注入章节意图，02b质检验证执行情况
- Delta JSON中的 `foreshadowing_ops` 字段自动更新伏笔追踪表的 `last_updated_chapter`

# Part G: 上下文衔接保障（P0/P1分级）

## G.1 问题
扩写时AI不翻看前文→前后不搭（伤势遗忘/位置跳跃/设定遗忘/关系不一致）。根本原因：03扩写时只有当前章细纲，没有前文上下文。

## G.2 解决方案：衔接包P0/P1分级
P0字段定义见上方"速查"。P1可选字段（不强制，缺失=02b警告但不阻断）：
- **[前章回顾]**：上一章核心事件（100字以内）+ 结尾状态
- **[必须延续的细节]**：前文提到、本章必须呼应的细节
- **[状态锁定]**：本章结束时须保持不变的状态
- **[闪回标注]**：本章是否含闪回/回忆/幻觉（涉及角色+起止标记+原因）
- **[本章新增状态]**：本章结束后角色/世界变化（含数值变更标注，须同步更新number_anchors）
- **[质感预算]**：指导03扩写的质感投放（感官投放+对话风格+比喻配额）
- **[跨章去重清单]**：由extract-used-patterns.js自动生成，手动维护时可跳过

> **统一跨章数据格式**：03/04/05所需跨章数据统一由衔接包"跨章去重清单"提供。角色对话模式追踪→追踪/角色状态.md（05写入，04/05读取）。比喻追踪→追踪/比喻追踪.md（05写入，03/05读取）。

## G.3 衔接包格式
完整输出格式见Part 6.2。精简原则：P0总字数≤300字，全部字段(P0+P1)≤500字。优先列"容易出错"信息（伤势/位置/关系）和"必须延续"信息（伏笔/设定/情绪）。

## G.4 衔接包生成规则
- 编写细纲时同步生成（不是额外步骤）
- 每章细纲完成后立即生成下一章衔接包，基于：当前章细纲+状态表+角色日记
- **链式生成**：上一章[本章新增状态]→下一章输入
- P0信息来源：角色状态→物理状态表+心理日记+00的power_system；禁止矛盾→状态表+设定；数值锁定→number_anchors；知识边界→前章[本章新增状态]

### G.4.1 Delta状态更新机制（V5.4新增 — 借鉴InkOS Delta模式）

> **核心问题**：当前衔接包是LLM全文生成，存在累积误差风险——一旦LLM在某章漏掉或改错一个状态字段，错误沿衔接包链传递。
> **解决方案**：LLM只输出本章变化（Delta），代码校验并合并到持久化状态文件。衔接包Markdown仍由02a生成（保持向后兼容），但增加结构化Delta JSON作为代码校验源。

**Delta生成规则**：
1. 02a生成衔接包时，同步输出 `chapter_delta` JSON块（见Part 6.2新增格式）
2. Delta只包含本章**发生变化**的字段——未变化的字段不输出（不是全量复制）
3. Delta字段与衔接包P0字段一一对应，但格式为JSON（便于脚本解析）

**Delta校验与合并流程**：
```
Delta合并流程(每章02a产出后执行):
├─ 1. 02a输出 chapter_delta JSON块（随衔接包一起输出）
├─ 2. 运行 node scripts/merge-delta.js --chapter=X --delta=chapter_delta.json --state=state/state.json --write
│   ⚠️ 必须加 --write 参数才会真正写入state.json，否则默认dry-run只报告不合并
├─ 3. 脚本校验:
│   ├─ 格式校验: JSON Schema合法性
│   ├─ 章节编号: 必须连续, 不可跳章
│   ├─ 角色ID: 必须存在于00设定JSON的character_matrix中
│   ├─ 伏笔ID: 必须存在于伏笔追踪表中（新增伏笔除外）
│   ├─ 数值合法性: 战力等级不可超过power_system上限, 数值不可倒退（如年龄变小）
│   └─ 生死状态: 已死亡角色不可"复活"（闪回标注除外）
├─ 4. 校验通过 → 合并到 state/state.json → 衔接包 sync_status=synced
├─ 5. 校验失败 → 不合并 → 衔接包 sync_status=degraded → 02b质检标记问题
└─ 6. 脚本输出: state/state.json(持久化全量状态) + state/state.md(Markdown投影,人可读)
```

> **与现有衔接包的关系**：衔接包Markdown仍然是02a的核心交付物，03a仍读衔接包。Delta JSON是**额外的代码校验层**——它不替代衔接包，而是为衔接包的状态字段提供代码级可靠性保障。当Delta校验失败时，衔接包Markdown仍然可用（降级模式），但02b会标记问题。

## G.5 衔接包与状态压缩配合（V5.4更新 — 脚本驱动+Delta）
- 状态压缩（每10章触发）：

```
状态压缩(每10章触发):
├─ 触发条件: 完成第10/20/30...章细纲后, 02b质检通过时触发
├─ 执行方式: 运行 node scripts/compress-handoff.js --chapters=1-10 --output=handoff_compressed_Ch1-10.json
├─ 脚本功能: 将Ch.1-10的衔接包P0字段提取合并, 去除已过期状态
├─ 输出: handoff_compressed_Ch1-10.json(≤500字)
└─ 扩写Ch.15时: 读取[handoff_compressed_Ch1-10.json]+[衔接包Ch.14]+[细纲Ch.15]

⚠️ 压缩由脚本执行, 不由LLM自动压缩。
```

- 衔接包（每章）：只含"上一章到本章"的增量信息，是状态压缩的"增量更新"

### G.5.1 state-degraded降级机制（V5.4新增 — 借鉴InkOS防状态污染）

> **核心原则**：正文可以重写，状态不能污染。当Delta校验失败时，防止错误状态向后续章节传播。

**降级触发条件**：
| 触发条件 | sync_status值 | 含义 | 03a行为 |
|---------|--------------|------|---------|
| Delta校验通过，合并成功 | `synced` | 状态已同步 | 正常读取衔接包P0字段 |
| Delta校验失败（格式/ID/数值错误） | `degraded` | 状态降级 | 读取P0但标注⚠️，依赖前章正文推断 |
| 02a未输出Delta JSON | `manual` | 手动模式 | 正常读取衔接包（向后兼容旧项目） |
| 脚本未执行/无Node环境 | `unverified` | 未验证 | 正常读取衔接包，02b标记"建议运行脚本" |

**降级处理规则**：
1. `degraded`状态的章节，02b质检报告中标记"状态降级"并列出校验失败原因
2. 03a读取`degraded`衔接包时，在输出头部标注"⚠️ 本章衔接包状态降级，以下P0字段可能不准确：[失败字段列表]"
3. 03a对`degraded`章节仅使用P0字段中校验通过的部分，失败字段从前章正文推断
4. 连续2章`degraded`→触发09多视角审稿师的异常触发机制（full审稿）
5. `degraded`状态不自动修复——须02a重新生成Delta或用户手动确认后标记为`manual`

# Part F2: 章节字数与结构控制

## F.1 原则
每章结构由**剧情需要**决定，非套固定模板。禁止机械套用"引入-推进-高潮-收束"。

## F.2 章节结构类型库（A-H）

| 类型 | 适用 | 结构 | 特点 |
|------|------|------|------|
| A·连续推进型 | 追逐/逃亡/连续战斗 | 场景1→2→3→4 | 全程推进，无明确高潮 |
| B·单点爆发型 | 对峙/揭秘/决战 | 铺垫30%→爆发50%→余波20% | 有明确高潮点 |
| C·日常流动型 | 日常/关系发展 | 场景1→2→3 | 情绪自然流动，无明显高潮 |
| D·信息揭示型 | 调查/探索 | 发现→分析→发现→串联→新方向 | 信息量递增 |
| E·情绪沉淀型 | 事件后消化/决策 | 回忆/反思→情绪处理→决定 | 内敛，动作少 |
| F·多线并进型 | 多视角/多地点 | 线A→线B→线A→线B | 视角切换制造张力 |
| G·倒叙/插叙型 | 揭示过去/补背景 | 当下→回忆→当下 | 时间跳跃，信息差 |
| H·起承转合型 | 完整小故事弧 | 起25%→承30%→转30%→合15% | 有完整弧光 |

## F.2.2 选择指南
- 剧情目标：推进主线→A/B/F | 揭示信息→D/G | 发展关系→C | 情绪沉淀→E | 完整事件→H
- 节奏：快→A/B/F | 慢→C/E | 中→D/H/G
- 卷中位置：卷初→B/A | 卷中→多样 | 卷末→B/H
- 01情感温度：高温→B/F | 中温→A/D | 低温→C/E | 未传递→自行判断

## F.3 字数控制
- 每章目标：4000字（±500，即3500-4500）
- 按场景重要性分配（非平均）：核心1200-1800 / 重要800-1200 / 过渡400-800 / 点缀200-400
- 场景数：3-5个。组合示例：核心×1+重要×2≈4000字

## F.4 细纲字数标注
每章必须包含[章节结构]（类型+理由+剧情目标+字数目标+章首钩子+爽点）和[场景规划]（每场景标注内容+重要性+字数+镜头指令）。

## F.4.5 镜头焦距指令（每场景必须标注）
- **[特写镜头]** 适用：情感爆发/关键动作/重大转折/爽点。放慢节奏，感官放大，每动作拆2-3步，占30-50%
- **[过场镜头]** 适用：赶路/进门/过渡。❌禁止环境铺陈（不写桌椅板凳/天气/路人），≤200字，3句内切换到核心动作
- **[中景镜头]** 适用：一般对话/信息交代。正常节奏，占15-25%
- 规则：每章特写1-2个，过场≥1个。禁止连续3特写或连续2过场。

## F.6 字数诊断速查
- **不足**：场景太少→拆分 | 描写太薄→加感官/心理/环境3层 | 对话太短→5-8轮+动作间隔 | 无呼吸空间→插入感官片段
- **过多**：场景太多→合并(≤5个) | 单场景过长→拆分(≤1500字) | 心理>15%→与动作绑定 | 环境>10%→服务情绪

## F.8 章节拆分规则
- 触发：>6000字 / 多个完整事件 / 大时间跨度 / 多地点 / 多高潮
- 原则：拆分点选自然断点（时间/地点/事件/情绪/信息），禁止在动作/对话/高潮中间硬切
- 拆分后每章3500-4500字，独立成立，前后章有钩子衔接
