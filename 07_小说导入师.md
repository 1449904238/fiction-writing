# 小说导入师 V2.0

> 融合自 oh-story-claudecode 项目的 story-import 功能。从已有正文反推为完整的写作工程项目结构，支持续写/精修/去AI味。
> **V2.0 重大更新**：① 接入V5.2机制（scene-trigger-map引用/INC 13要素反向提取/用户情绪锚点提取/02a-02b接口）；② JSON Schema对齐协议（与00设定架构师完全兼容）；③ 信号词扫描改进（置信度分级+上下文验证+批量处理）；④ 导入后自动生成首批V5.2衔接包。

## 使用流程

### 第一步：确定导入范围

| 信息 | 说明 |
|------|------|
| 已有正文 | 多少章？每章约多少字？ |
| 保存位置 | 正文文件路径 |
| 当前进展 | 写到什么阶段了？有无细纲/设定/大纲？ |
| 目标 | 续写/精修/去AI味/全流程 |
| 题材方向 | 玄幻/都市/悬疑/言情/科幻...（决定JSON模块启用） |

### 第二步：扫描正文 → 提取结构化数据

逐章扫描已有正文，提取以下信息：

```
【章节信息表】

| 章节 | 字数 | 核心事件 | 登场角色 | 关键物品 | 伏笔/悬念 | INC要素命中 |
|------|:----:|----------|----------|----------|-----------|:-----------:|
| Ch.1 | 4000 | 师父跑了 | 沈夜,赵叔 | 铜钱 | 师父去哪了 | ①②⑦ |
| Ch.2 | 4200 | 第一个案子 | 沈夜,柳如烟 | 镜子 | 画皮怪 | ①③⑤⑨ |
| ... | ... | ... | ... | ... | ... | ... |
```

> **批量处理方案（V2.0新增）**：每20章为一批进行扫描，跨章合并提取结果。批间产出中间摘要，最终汇总为完整的章节信息表。避免逐章扫描导致的上下文丢失和跨章伏笔遗漏。

### 第三步：重建设定文档 + JSON Schema对齐

从正文中提取并重建 `00_产出_设定文档.md`。**导入产出的JSON元数据必须与00设定架构师的Schema完全兼容**（见下方"Schema对齐协议"）。

```
【设定提取模板】

1. 世界观规则
   ├─ 从正文提取：角色提到了什么规则？
   ├─ 从事件提取：发生了什么说明规则的？
   └─ 待补充：有哪些规则已暗示但未明说的？

2. 角色档案
   ├─ 角色名：（从正文提取）
   ├─ 身份：（从对话/叙述提取）
   ├─ 性格：（从行为/对话提取）
   ├─ 能力/金手指：（从表现提取）
   ├─ life_chain（V2.0新增）：从正文反推角色生平关键事件链
   └─ 关系：（从互动线提取）

3. 核心冲突
   ├─ 主要冲突：（从主线提取）
   ├─ 次要冲突：（从支线提取）
   └─ 未解决的冲突：（作为伏笔保留）

4. JSON元数据（V2.0 — 严格对齐00 Schema）
   ├─ novel_meta: { title, genre, core_tags, emotion_hook, version, active_modules }
   ├─ characters: [{ name, role, character_position, core_tags, life_chain, ... }]
   ├─ relationships: [{ pair, surface, deep, evolution_path }]
   ├─ power_system: { ... } （非玄幻题材降级跳过）
   ├─ number_anchors: { timeline, character_numbers, distances, counts }
   └─ anti_patterns: { forbidden, required }
```

### Schema对齐协议（V2.0新增）

> **核心约束**：导入产出的JSON元数据必须能被02a/03a等下游技能直接解析，字段名/结构/枚举值与00设定架构师的Schema定义完全一致。

#### 必填字段清单（与00 Schema逐字段对齐）

| 顶层字段 | 来源 | 必填 | 说明 |
|----------|------|:----:|------|
| `novel_meta` | 从正文推断 | ✅ | 含 title/genre/core_tags/emotion_hook/version/active_modules |
| `characters` | 从正文提取 | ✅ | 至少1个角色，含 name/role/character_position/core_tags |
| `relationships` | 从互动推断 | ✅ | 至少1组关系，含 pair/surface/deep/evolution_path |
| `power_system` | 从战斗描写提取 | ⚠️ | 玄幻/科幻必填，其他题材降级跳过 |
| `number_anchors` | 从正文提取 | ✅ | timeline/character_numbers(含status生死)/distances/counts |
| `anti_patterns` | 从正文识别 | ✅ | forbidden/required 两个列表 |

#### 字段缺失降级策略

| 场景 | 降级处理 | active_modules 设置 |
|------|----------|-------------------|
| **非玄幻题材**（都市/言情/悬疑等） | 跳过 power_system，用 alternative_module 替代 | `["number_anchors"]`，`alternative_module` 设为"社会关系矩阵"/"证据链追踪表"等 |
| **正文无战斗描写** | power_system 字段设为 `null`，active_modules 不含"power_system" | `["number_anchors"]` |
| **角色 life_chain 无法完整反推** | 已知事件填入对应 event 字段，未知事件标注 `"[待补充：正文未提及]"` | — |
| **角色 gold_finger 不明确** | ability 填入正文中的表现描述，cost 填 `"[待确认]"`，growth_stages 按卷数占位 | — |
| **number_anchors 角色生死状态不明确** | 默认 status="存活"，标注 `"[待确认]"` | — |
| **正文过短（<5章）** | 关系/数值/伏笔字段允许大面积 `[待补充]`，提示用户在02a阶段完善 | — |

#### JSON 语法校验清单

导入完成后必须执行以下校验：
- [ ] JSON 语法完整可解析（无缺逗号/引号/括号）
- [ ] novel_meta.title / genre / core_tags / emotion_hook / version 全部填写
- [ ] characters 数组至少1个角色，role 枚举值仅为"主角"/"配角"/"反派"
- [ ] number_anchors.character_numbers 中每个角色的 status 为"存活"或"已死亡"
- [ ] anti_patterns.forbidden 和 required 均为数组
- [ ] **同时导出独立文件** `设定/00_产出_JSON元数据.json`（与文档内嵌JSON内容完全一致）
- [ ] active_modules 与实际启用的模块一致（非玄幻不含 power_system）

### 第四步：正文反推启发式规则（V2.0改进 — 信号词扫描+置信度分级+上下文验证）

#### 1. 伏笔识别启发式（V2.0改进）

扫描正文中的以下信号，识别可能被埋下的伏笔：

- **信号词扫描**："奇怪的是/不对劲/似乎/好像/不知道为什么/莫名/隐约"
- **未解释的细节**：提及但未在3章内解释的物品/行为/对话
- **角色欲言又止**："算了/没什么/以后再说/不是现在"
- **读者疑问点**：角色做了不合常理的事但未解释原因
- **反常强调**：叙事中对某个细节给了异常多的笔墨但看似无关紧要

##### 置信度分级标注（V2.0新增）

每个候选伏笔标注置信度等级：

| 置信度 | 判定条件 | 标记 | 处理方式 |
|:------:|----------|------|----------|
| **高** | 信号词 + 后文有明确回收线索（"原来/果然/难怪/怪不得/这就说得通了"） | `置信度:高` | 直接进入伏笔追踪表，标记为 ACTIVE |
| **中** | 信号词 + 上下文有间接呼应（同一物品/角色在后续章节再次出现且态度变化） | `置信度:中` | 进入伏笔追踪表，标记为"待确认" |
| **低** | 仅凭信号词出现，无上下文呼应 | `置信度:低` | 标记为"候选"，需用户确认后再入表 |

##### 上下文验证机制（V2.0新增）

检测到信号词后，**必须执行上下文验证**：

1. **前100字扫描**：检查信号词前100字是否有铺垫性描述（物品特写/角色异常行为/环境暗示）
2. **后100字扫描**：检查信号词后100字是否有回收性表达（"原来/果然/难怪/怪不得/这就说得通了/所以这就是"）
3. **跨章验证**：在后续5章范围内搜索同一物品/角色/线索是否再次出现
4. **降级规则**：单独信号词（无上下文呼应）的置信度从"候选伏笔"降为"待确认"，不自动入表

```
【信号词扫描 + 上下文验证流程】

Step 1: 扫描信号词 → 命中
Step 2: 提取信号词位置（章号+行号）
Step 3: 前100字扫描 → 有铺垫？→ 是=+1分
Step 4: 后100字扫描 → 有回收线索？→ 是=+2分（直接升级为"高"）
Step 5: 跨章验证（后续5章）→ 同一线索再现？→ 是=+1分
Step 6: 置信度判定：
        总分≥3 → 高（直接入表，ACTIVE）
        总分=2 → 中（入表，待确认）
        总分≤1 → 低（候选，需用户确认）
```

对每个候选伏笔，标注：发现章号 | 相关角色 | 推测类型（物品伏笔/对话伏笔/行为伏笔/细节伏笔）| 置信度（高/中/低）| 上下文验证得分

#### 2. 设定反推启发式
- **世界观**：扫描"在这个世界/XX的规矩是/从来都是/众所周知"等规则性表达
- **力量体系**：扫描战斗描写中的能力表现+限制条件+修炼方式 → 填入 power_system
- **角色关系**：扫描对话中的称谓/态度/历史提及，构建关系图谱 → 填入 relationships
- **地理设定**：扫描地名+距离+方位描述，构建地图 → 填入 number_anchors.distances
- **社会结构**：扫描组织/势力/阶级/职业相关的描述

#### 3. 伏笔回收识别
- 扫描"原来/果然/所以这就是/难怪/怪不得/这就说得通了"等揭示性表达
- 匹配前文伏笔候选列表
- 标记为"已回收"并记录回收章号
- 未回收的伏笔标注为ACTIVE，进入伏笔追踪表

#### 4. 提取置信度标注
每个提取结果标注置信度：
- **高**：正文中有明确文本证据
- **中**：通过上下文推断，证据较充分
- **低**：仅凭信号词推测，需用户确认

### 第五步：V5.2机制接入 — INC 13要素反向提取（V2.0新增）

> **INC = Interesting New Content**。从已有正文中反向识别INC 13要素，为后续大纲重建和续写提供"哪些维度已有内容"的基线。

对已有正文逐卷（或每10章一批）扫描，识别以下13项要素的命中情况：

```
【INC 13要素 — 反向提取表】

├─ ① 新信息：正文向读者揭示了什么此前未知的信息？
├─ ② 新冲突：引入了什么新的核心矛盾？
├─ ③ 新角色/新面相：有新角色登场，或已有角色展现新侧面？
├─ ④ 新场景：带读者去了什么新地点/新世界？
├─ ⑤ 新能力/新手段：主角获得了什么新能力/新资源/新手段？
├─ ⑥ 新关系：角色关系发生了什么新变化（结盟/反目/暧昧/背叛）？
├─ ⑦ 新悬念：埋设了什么让读者想知道答案的新谜题？
├─ ⑧ 新反转：有什么出人意料的剧情转折？
├─ ⑨ 新情绪：让读者体验了什么此前未有的情绪？
├─ ⑩ 新代价：角色为目标付出了什么不可逆的代价？
├─ ⑪ 新选择：主角面临了什么两难抉择？
├─ ⑫ 新障碍：出现了什么此前不存在的阻力量？
└─ ⑬ 新认知：角色/读者对世界/自身有了什么新理解？

输出格式：
| 卷/章范围 | 命中要素编号 | 对应正文证据 | 续写建议 |
|-----------|:----------:|-------------|----------|
| Ch.1-10 | ①②⑦⑨ | Ch.1师父失踪(新悬念), Ch.3画皮怪(新信息) | ⑧反转⑩代价维度薄弱，续写时补充 |
```

**续写指导**：
- 命中<4项的卷/批次 → 标记"内容维度薄弱"，续写时优先补充
- 相邻批次INC命中维度≥3项重叠 → 节奏雷同预警，续写时需换维度
- 全书无⑧新反转 → 续写时必须设计反转
- 全书无⑩新代价 → 续写时需增加重量感

### 第六步：V5.2机制接入 — 用户情绪锚点提取（V2.0新增）

> 从已有正文中识别情绪交付模式，生成建议锚点，供02a细纲生成时使用。

**提取流程**：

1. **情绪模式扫描**：逐章识别正文交付的核心情绪（爽/虐/怕/甜/思/燃/释然/紧张/期待...）
2. **情绪强度评估**：根据正文篇幅占比+描写密度评估情绪强度（强/中/弱）
3. **情绪锚点识别**：找出承载核心情绪的物理物件/动作/场景（如"铜钱→师父的牵挂→虐"）
4. **建议锚点生成**：为后续章节生成建议情绪锚点

```
【用户情绪锚点 — 提取报告】

已有正文的情绪交付模式：
| 章节范围 | 核心情绪 | 强度 | 承载锚点（物理物件/动作） | 效果评估 |
|-----------|:--------:|:----:|--------------------------|----------|
| Ch.1-3 | 虐+期待 | 强 | 铜钱（师父遗物）/ 师父背影 | 有效：悬念驱动追读 |
| Ch.4-6 | 紧张+爽 | 中 | 镜子碎裂 / 画皮怪现形 | 有效：案件推理驱动 |
| Ch.7-10 | 燃+释然 | 强 | 沈夜拔刀 / 真相揭露 | 有效：高潮释放 |

建议后续章节情绪锚点：
├─ Ch.11+：建议交付"震惊+期待"（新角色登场反转）
├─ 锚点建议：[新角色的信物] → 承载"未知"的恐惧与好奇
└─ 注意：已有正文"虐"的情绪占比较高，续写时适当增加"爽"的比例以平衡
```

> **与02a的接口**：此情绪锚点报告作为02a细纲生成时"用户情绪锚点"字段的输入参考。02a在生成新章节细纲时，须参考已有情绪模式，避免情绪单一化。

### 第八步：重建细纲与V5.2衔接包生成（V2.0新增）

为已有章节补写细纲，为新章节编写细纲：

#### V5.2衔接包自动生成（V2.0新增）

> 导入完成后，为已有正文的**最后一章**自动生成首批衔接包，供02a续写时直接使用。衔接包格式严格遵循02a的P0/P1分级标准。

从已有正文提取以下P0必须字段（限300字以内）：

```
【衔接包 — 第X章（已有正文末章）】（V5.2.0 P0/P1分级格式）

━━━ P0必须字段（限300字以内）━━━

[角色当前状态] 【P0】
| 角色 | 位置 | 身体状态 | 心理状态 | 关键持有物 | 战力等级 | 生死状态 |
| {从正文末章提取} | ... | ... | ... | ... | ... | 存活/已死亡 |

[禁止矛盾] 【P0】
├─ ❌ {从正文提取的已确立约束}
├─ ❌ {角色已受伤的部位不能正常活动}
├─ ❌ {已死亡角色不能以活人状态出现}
└─ ❌ {已确立的角色习惯不能突然改变}

[数值锁定] 【P0】
├─ 角色年龄：{从正文提取}（须与 number_anchors.character_numbers 一致）
├─ 修炼计数/能力等级：{从正文提取}
├─ 地理距离：{从正文提取}
├─ 物品数量：{从正文提取}
├─ 时间线：当前=故事第X天/年
└─ 生死状态：{已死亡角色}={死亡章号}→ ⚠️续写如涉及须为闪回

[角色知识边界] 【P0】
├─ {角色A}：已知{...} | 未知{...}
├─ {角色B}：已知{...} | 未知{...}
└─ ⚠️ {信息差标注}

━━━ P1可选字段 ━━━
```
