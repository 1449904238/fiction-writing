## 四审流程

### 第1关：编辑视角（复用04精修师）

调用 `04_小说正文精修师.md` 的标准精修流程，重点关注：

- 潜台词打磨是否到位
- 金句提炼是否足够
- 情感留白是否自然
- 感官锚点是否深度
- 标点规则是否合规（省略号≤5/破折号≤4/感叹号≤15）

**产出**：质感评分（1-10）+ 精修标记

### 第2关：AI鉴定师视角（复用05去AI味精修师）

调用 `05_去AI味精修师.md` 的11 Gate系统检测：

- Gate 1-11 逐项通过/不通过
- 综合AI味评分
- 建议修改优先级（P0/P1/P2）

**产出**：AI味评分 + 11 Gate检测报告

### 第3关：读者视角

```
【读者视角审阅模板】

1. 第一印象（前500字）
   ├─ 第1句是否抓人？ （是/否）
   ├─ 前200字是否有冲突/悬念？ （是/否）
   ├─ 前500字是否有爽点/钩子？ （是/否）
   └─ 整体：想继续读吗？ （很想/还行/不想）

2. 阅读体验
   ├─ 节奏是否舒服？ （快/适中/慢）
   ├─ 有没有读不下去的地方？ （标注位置）
   ├─ 有没有特别爽/感动/紧张的段落？ （标注位置）
   └─ 整体：读起来累吗？ （轻松/适中/吃力）

3. 角色代入感
   ├─ 你喜欢主角吗？ （非常/一般/不喜欢）
   ├─ 你能理解主角的行为吗？ （总是/有时/从不）
   ├─ 有喜欢的配角吗？ （谁？为什么？）
   └─ 反派有压迫感吗？ （有/一般/没有）

4. 情绪反应
   ├─ 这一章让你笑了吗？
   ├─ 让你紧张了吗？
   ├─ 让你感动了吗？
   ├─ 让你生气了吗？
   └─ 整体：你感受到了什么情绪？

5. 扣分项
   ├─ 有哪些地方出戏了？
   ├─ 有哪些地方看不懂？
   ├─ 有哪些地方觉得假/不合理？
   └─ 有哪些地方觉得无聊/想跳过？

6. 信息密度评估（情节含金量）
   ├─ 是否存在"文字优美但读完发现什么都没发生"的段落？（标注位置）
   ├─ 主角是否在非冲突场景做了大量微动作但信息零推进？
   ├─ 是否有"一句话能概括但写了300字"的注水段落？
   └─ 整章读完，读者获得了多少"新信息"？（多/一般/少）
```

读者视角评分标准（各维度0-100分）：
- 第一印象（前300字是否抓住注意力）：≥80分=想继续读 / 60-79=还行 / <60=想弃文
- 阅读体验（流畅度、节奏感）：≥80分=一口气读完 / 60-79=偶尔走神 / <60=多次想弃
- 角色代入（是否能共鸣）：≥80分=强烈共鸣 / 60-79=有点感觉 / <60=无感
- 情绪反应（是否有情感波动）：≥80分=多次情绪波动 / 60-79=偶尔波动 / <60=无波动
- 信息密度（是否觉得每段都有价值）：≥80分=全程有价值 / 60-79=有水段 / <60=大量水段
- 扣分项（每出现1项扣5分）：错别字/逻辑矛盾/人设崩塌/节奏拖沓/对话注水

### 第4关：商业化评估

```
【商业化评估模板】

1. 市场适配度
   ├─ 类型：是否符合主流平台的热门类型？
   ├─ 节奏：是否符合快节奏/慢热/中等？
   ├─ 爽点密度：每多少字一个爽点？
   └─ 更新兼容性：适合日更吗？（4000字/章）

2. 留存力评估
   ├─ 读者看完这章，想看下一章吗？
   ├─ 钩子质量：章末钩子是否有效？
   ├─ 主线清晰度：读者知道故事要往哪走吗？
   └─ 期待感：读者在期待什么？

3. 差异化评估
   ├─ 与同类作品相比，有什么独特卖点？
   ├─ 有哪些元素是"别人写过很多次"的？
   └─ 有哪些元素可能成为"出圈"的理由？

4. 平台适配
   ├─ 字数：是否符合平台要求？
   ├─ 敏感内容：是否有审核风险？
   └─ 推荐友好度：前500字是否符合平台推荐算法？
```

### 第5关：一致性审计（V2.1新增 — 数值/战力/知识边界全书级抽样）

> **定位**：前4关聚焦"写得好不好"，本关聚焦"前后对不对"。在全书或大段章节完成后执行，对照00设定的 `power_system` 和 `number_anchors` 做抽样审计。

```
第5关：一致性审计（抽样检查，非逐章）

1. 数值一致性抽样
   ├─ 随机抽取3个角色，核对正文中出现的年龄/修炼枚数/骨骼类型是否与 number_anchors 一致
   ├─ 随机抽取3段距离描写，核对是否与 number_anchors.distances 一致
   ├─ 核对时间线：故事开始至今的时间流逝是否与 number_anchors.timeline 自洽
   └─ 核对物品数量：关键物品的数量/大小是否前后一致

2. 战力一致性抽样
   ├─ 随机抽取3个战斗场景，核对角色表现是否在 power_system 矩阵的 ability_ceiling 内
   ├─ 越级战斗是否有设定依据？（金手指/道具/合击/代价）
   ├─ 旁白描述的危险等级与实际战斗难度是否匹配？
   └─ 战力波动是否有逻辑？（受伤/突破/消耗）

3. 生死状态审计
   ├─ 全书所有"已死亡"角色是否在后续正文中以活人状态出现？（非闪回）
   ├─ 闪回段落是否用了明确的时态/视角切换标记？
   └─ 死亡角色的死亡时间/死因是否与 number_anchors 一致？

4. 知识边界抽样
   ├─ 随机抽取3个信息差场景，核对角色是否说了不该知道的信息
   ├─ 角色对秘密的反应是否有前文信息传递场景支撑？
   └─ 信息差解除的时机是否合理？

输出：一致性审计报告（findings格式，severity S1-S4）
```

### 多平台评分标准（v2.0 新增）

**审稿时按目标平台加载对应 rubric，无法确定平台时使用通用网文标准：**

| 平台 | 核心评分维度 | 权重侧重 |
|------|------------|---------|
| **番茄小说** | 强开局、强冲突、高频爽点/情绪反馈、低理解门槛 | 爽感密度30%+节奏20%+钩子20%+代入15%+文笔15% |
| **起点中文网** | 设定自洽、升级路径清晰、长线期待、世界观承载力 | 设定30%+结构25%+爽感20%+文笔15%+钩子10% |
| **知乎盐言** | 短篇钩子密集、反转密度高、情绪兑现快、信息差推进 | 反转30%+钩子25%+情绪25%+文笔20% |
| **通用网文** | 综合评估，各维度均衡 | 均等分配 |

**番茄特供审稿要点**（当目标平台为番茄时额外检查）：
1. **黄金三章法则**：前3章必须在3000字内完成"建立人设→制造冲突→给出爽点"三步
2. **爽感密度**：每2000字至少一个情绪反馈点（爽点/笑点/紧张点/泪点）
3. **低理解门槛**：核心设定能否用一句话说清？读者前500字能否理解主角处境？
4. **追读钩子**：章末是否留了"不点下一章就难受"的钩子？连续3章钩子类型不重复
5. **章节定位与张弛**：高压/推进章配齐钩子+爽点+悬念；低压/关系/修炼章允许弱钩子，但每章都给读者一个往下看的理由

## 三档审稿模式（v2.0 新增）

### 模式选择

| 模式 | 触发方式 | 视角数 | 执行方式 | 适用场景 |
|------|---------|:------:|---------|---------|
| **full** | `/review_full` | 4 | 并行 spawn 4 个 Agent | 重要章节/卷末/质量要求高时 |
| **lean** | `/review_lean` | 2 | 主线程执行编辑+读者 | 日常章节/快速审稿 |
| **solo**（默认） | `/review_solo` 或无参数 | 4 | 主线程串行执行全部 4 关 | 无 subagent 支持时 |

### 降级链

```
full（4 Agent 并行）
  ↓ 条件：环境不支持 subagent / spawn 失败 / 超时
lean（2 Agent 精简）
  ↓ 条件：精简 Agent 也不可用
solo（主线程串行，等同于 V1.0 模式）
```

**每层降级必须报告**：
```
[降级] full → lean：原因 — 当前平台不支持 subagent spawn
[降级] lean → solo：原因 — Agent 配置文件不可读
```

### Full模式Agent协同规范（V2.1补充）
**通信协议**：
- 每个Agent独立运行，产出独立的Findings列表（YAML格式）
- 不需要Agent间实时通信，各自基于同一章节正文独立评审
**Findings合并机制**：
1. 收集4个Agent的Findings列表
2. 按 severity 排序：S1(致命)>S2(严重)>S3(建议)>S4(提示)
3. 同一位置的多个Agent findings合并为一条，标注来源（如"[编辑+读者]"）
4. 冲突处理：S1/S2级finding任意Agent提出即标记为"需修改"；S3/S4级按多数票决定
**综合评分公式**：
- 综合得分 = 编辑视角×30% + AI鉴定×25% + 读者视角×25% + 商业化×20%
- 各视角满分100分，综合得分满分100分
- 综合得分≥85为优秀，70-84为合格，<70需返工

### 确定性脚本预检（所有模式共用）

审查前自动运行 4 个确定性脚本，报告作为 baseline 注入各视角：

```bash
node scripts/check-ai-patterns.js --check <正文文件>
node scripts/check-degeneration.js --check <正文文件>
node scripts/normalize-punctuation.js --check <正文文件>
node scripts/check-consistency.js --meta=<JSON元数据> <正文文件>
```

脚本发现 ≥1 blocking 级 finding 时，**先修复脚本发现再进入 LLM 审查**（脚本问题是确定性的，不值得浪费 LLM 上下文）。`check-consistency.js` 的报告直接作为第5关一致性审计的 baseline。

## 统一 Findings Schema（v2.0 新增）

所有视角的输出必须使用统一的 Findings 格式（YAML）：

```yaml
findings:
  - severity: S1          # S1=必须修复 / S2=强烈建议 / S3=建议优化 / S4=锦上添花
    category: prose        # plot|character|prose|consistency|commercial|consistency_audit  # V2.1新增 consistency_audit
    location: Ch.5 L120-135
    evidence: "他看着远方，心中涌起一股莫名的悲伤"
    issue: "AI味：'看着'+'心中涌起'+'莫名的'三重叠加"
    fix: "改为具体动作：'他蹲下身，捡起一块石头抛了抛'"
    gate: G1               # 对应 Gate 编号（仅 AI 味类）
    source: 编辑视角        # 产出视角名称
```

**severity 等级定义**：
| 等级 | 含义 | 处理要求 |
|------|------|---------|
| S1 | 必须修复 | 不修复不定稿，blocking 级问题 |
| S2 | 强烈建议 | 严重影响阅读体验 |
| S3 | 建议优化 | 可以提升品质 |
| S4 | 锦上添花 | 有时间再改 |

## 流水线契约

- 第1-2关**直接复用**04和05技能，不做重复规则定义
- 第3-4关为**新增视角**，不取代04/05的职责
- 审稿频率：建议每5章或每卷完成时执行
- 综合评分<6/10时，建议回到对应步骤重新修改
- **v2.0 新增**：所有视角输出必须使用统一 Findings Schema（YAML）
- **v2.0 新增**：审查前先跑确定性脚本预检，blocking finding 清零后再 LLM 审查
- **v2.0 新增**：支持 full/lean/solo 三档模式，solo 为默认
- **v2.0 新增**：按目标平台加载评分 rubric（番茄/起点/知乎/通用）
