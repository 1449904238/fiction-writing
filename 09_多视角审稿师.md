# 多视角审稿师 V5.3

> **V5.3 更新**：审稿触发机制从"每3章必执行full审稿"改为**分级触发**——每1章脚本检测（不启动LLM）/每5章lean模式审稿（2关）/每10章卷末full模式审稿（4Agent+第5关一致性审计）/异常触发（03b连续2章<70分→full；脚本blocking→lean；用户手动调用）。新增审稿疲劳防护（连续3次full>85分→降级lean）与lean结果累积机制（full审稿时做趋势分析）。4Agent协同规范保持不变。

> **V5.2 更新**：审稿频率从"每5章"调整为"每3章必执行一次"——第3/6/9/12...章自动触发审稿。同步落地4个独立 Agent（`.claude/agents/` 下的 story-editor / ai-detector / reader / commercial），第3-4关的"读者视角 Agent""商业化评估 Agent"占位符替换为实际文件引用（reader-agent.md / commercial-agent.md）。V5.2新增：Agent评分统一0-100分制，reader爽感权重5%→15%。

> **V2.1 更新**：新增第5关一致性审计——数值/战力/生死/知识边界全书级抽样检测，对照00设定power_system和number_anchors。

> **V2.0 更新**：融合 oh-story-claudecode 的多平台 rubric 体系。新增番茄/起点/知乎三平台评分标准 fallback，确定性脚本预检引用。
>
> 融合自 oh-story-claudecode 项目的 story-review 系统。用四个视角全面审阅小说正文，输出带优先级的修改建议。

## 分级触发机制（V5.3 新增）

> 替代 V5.2 的"每3章必执行full审稿"。核心思路：高频轻检、低频深审、异常即触发。

### 触发频率定义

| 频率 | 触发条件 | 执行内容 | 是否启动LLM |
|------|---------|---------|:-----------:|
| **每1章** | 每章写完自动 | 确定性脚本检测（check-ai-patterns / check-degeneration / normalize-punctuation / check-consistency） | 否（纯脚本） |
| **每5章** | 第5/10/15/20...章 | lean模式审稿（主线程执行编辑视角+读者视角2关） | 是（2关） |
| **每10章（卷末）** | 第10/20/30...章或卷末 | full模式审稿（4Agent并行4关+第5关一致性审计） | 是（4Agent+1审计） |
| **异常触发** | 见下方"异常触发条件" | 按异常类型选择lean或full | 是 |

> 注：第10章同时满足"每5章"和"每10章"时，执行更高级别的full模式（不重复执行lean）。卷末无论是否为第10/20章整数倍，均执行full模式+第5关一致性审计。

### 异常触发条件

异常触发独立于固定频率，满足任一条件即立即启动对应级别审稿：

| 异常类型 | 触发条件 | 执行级别 | 说明 |
|---------|---------|---------|------|
| 质量滑坡 | 03b质检连续2章综合评分<70分 | full | 立即4Agent并行审稿，定位质量下滑根因 |
| 脚本拦截 | 确定性脚本发现blocking级问题 | lean | 脚本问题确定性高，lean快速复核即可 |
| 用户手动 | 用户调用 `/review_full` `/review_lean` `/review_solo` | 按用户指定 | 手动调用优先级最高，覆盖一切自动判定 |

> 异常触发的审稿**不重置**固定频率计数器（如第7章因异常触发full，第10章仍按计划执行full）。

### 审稿疲劳防护（V5.3 新增）

> 防止高质量章节反复消耗full审稿的4Agent上下文成本。

```
疲劳防护规则：
├─ 追踪最近3次full模式审稿的综合得分
├─ 若连续3次full审稿综合得分均 > 85分（优秀线）
│   └─ 下一次原计划full的触发点自动降级为lean模式
│   └─ 降级时输出提示：[疲劳防护] 连续3次full>85分，本次自动降级lean
├─ lean审稿后若综合得分 < 80分
│   └─ 立即补执行一次full审稿（撤销降级，回归深度审查）
└─ 疲劳防护状态在卷末full审稿后重置（每卷重新计数）
```

**设计意图**：当连续多章质量稳定优秀时，full审稿的边际收益递减，降级为lean可节省4Agent上下文开销；一旦lean发现质量回落，立即回升到full，确保不漏检。

### lean结果累积机制（V5.3 新增）

> lean审稿不是"用完即弃"，其结果累积存储，供full审稿做趋势分析。

```
累积机制：
├─ 每次lean审稿的Findings列表 + 综合得分 → 写入审稿累积库（review_accumulation.md）
├─ 累积字段：章节号 / 模式 / 综合得分 / 分项得分 / blocking-advisory-info finding计数 / 重复finding标记
├─ full审稿启动时，先读取累积库中本卷所有lean结果
│   └─ 做趋势分析：得分曲线 / 高频问题类别 / 跨章重复finding / 是否恶化
│   └─ 趋势分析报告作为full审稿的附加输入注入4个Agent
└─ 趋势分析重点关注：
    ├─ 同类问题是否跨5章窗口反复出现？（说明未根治）
    ├─ 综合得分是否呈下降趋势？（说明质量滑坡，需深查）
    └─ 某个Agent视角的得分是否持续偏低？（说明该维度有系统性问题）
```

> **与第5关一致性审计的关系**：累积库中的lean结果为第5关（一致性审计）提供跨章数据支撑——lean审稿中标记的"consistency"类finding会在full审稿时被第5关重点复核。

## 四审流程

### 职责边界矩阵（V5.3.1 新增 — 正交检测域划分）

> 四个Agent的检测域必须正交，避免重复评估同一维度造成评分偏差和资源浪费。

| Agent | 核心问题 | 独占检测域 | 不再检测 |
|-------|---------|-----------|---------|
| 编辑 | 文字技艺达标吗？ | 潜台词/比喻质量/句法/标点/记忆点 | →爽感归商业,代入感归读者,节奏归读者 |
| AI鉴定 | 像AI写的吗？ | 11Gate纯AI味特征 | →质感归编辑,体验归读者 |
| 读者 | 读者爽/代入吗？ | 代入感/节奏体感/出戏点/情绪反应/弃文冲动 | →爽点密度归商业,信息投放归商业 |
| 商业 | 卖得出去吗？ | 留存力/追更动力/差异化卖点/平台适配/对标 | →代入感归读者,质感归编辑 |

> 每个Agent prompt中增加"你不评估X（归其他Agent）"的负面约束。

### 第1关：编辑视角（复用04精修师）

调用 `04_小说正文精修师.md` 的标准精修流程，重点关注：

- 潜台词打磨是否到位
- 金句提炼是否足够
- 情感工艺是否自然（技术层面：Show/Tell比例、情绪词配额、留白节奏）
- 标点规则是否合规（省略号≤5/破折号≤4/感叹号≤15）

> 边界约束（V5.3.1）：编辑不评估"代入感"（归读者）、"爽感"（归商业）、"节奏体感"（归读者）。只管文字技艺是否达标。

**产出**：质感评分（1-10）+ 精修标记

### 第2关：AI鉴定师视角（复用05去AI味精修师）

调用 `05_去AI味精修师.md` 的11 Gate系统检测：

- Gate 1-11 逐项通过/不通过
- 综合AI味评分
- 建议修改优先级（P0/P1/P2）

**产出**：AI味评分 + 11 Gate检测报告

### 第3关：读者视角

> V5.1：full 模式下由独立 Agent `.claude/agents/reader-agent.md` 执行，6 维度审查（代入感/节奏感/角色辨识度/情绪共鸣/悬念期待/阅读流畅度），综合评分权重 25%。以下模板为 solo/lean 模式主线程执行时的参照。
>
> V5.3.1 边界调整：删除"爽感满足"（归商业Agent）、删除"信息获取"（归商业Agent的信息投放），聚焦读者体验维度。

```
【读者视角审阅模板】

1. 第一印象（前500字）
   ├─ 第1句是否抓人？ （是/否）
   ├─ 前200字是否有冲突/悬念？ （是/否）
   ├─ 前500字是否有爽点/钩子？ （是/否）
   └─ 整体：想继续读吗？ （很想/还行/不想）

2. 阅读体验
   ├─ 节奏是否舒服？ （快/适中/慢）
   ├─ 有没有读不下去的地方？ （标注位置）
   ├─ 有没有特别爽/感动/紧张的段落？ （标注位置）
   └─ 整体：读起来累吗？ （轻松/适中/吃力）

3. 角色代入感
   ├─ 你喜欢主角吗？ （非常/一般/不喜欢）
   ├─ 你能理解主角的行为吗？ （总是/有时/从不）
   ├─ 有喜欢的配角吗？ （谁？为什么？）
   └─ 反派有压迫感吗？ （有/一般/没有）

4. 情绪反应
   ├─ 这一章让你笑了吗？
   ├─ 让你紧张了吗？
   ├─ 让你感动了吗？
   ├─ 让你生气了吗？
   └─ 整体：你感受到了什么情绪？

5. 扣分项
   ├─ 有哪些地方出戏了？
   ├─ 有哪些地方看不懂？
   ├─ 有哪些地方觉得假/不合理？
   └─ 有哪些地方觉得无聊/想跳过？

> 注：信息密度评估已移至商业Agent（信息投放节奏维度），读者不评估"信息获取"。
```

读者视角评分标准（各维度0-100分）：
- 第一印象（前300字是否抓住注意力）：≥80分=想继续读 / 60-79=还行 / <60=想弃文
- 阅读体验（流畅度、节奏感）：≥80分=一口气读完 / 60-79=偶尔走神 / <60=多次想弃
- 角色代入（是否能共鸣）：≥80分=强烈共鸣 / 60-79=有点感觉 / <60=无感
- 情绪反应（是否有情感波动）：≥80分=多次情绪波动 / 60-79=偶尔波动 / <60=无波动
- 扣分项（每出现1项扣5分）：错别字/逻辑矛盾/人设崩塌/节奏拖沓/对话注水

> 注：信息密度评分维度已移至商业Agent，读者不评估。

### 第4关：商业化评估

> V5.1：full 模式下由独立 Agent `.claude/agents/commercial-agent.md` 执行，7 维度审查（开篇吸引力/章末钩子/爽点密度/信息投放节奏/对标差异化/节奏控制/完读率预判），综合评分权重 20%，额外输出 P0/P1/P2 商业修改建议优先级排序。以下模板为 solo/lean 模式主线程执行时的参照。
>
> V5.3.1 边界确认："爽点密度"为商业Agent独占检测域（读者不再评估爽感）。商业Agent不评估"代入感"（归读者）、"质感"（归编辑）。

```
【商业化评估模板】

1. 市场适配度
   ├─ 类型：是否符合主流平台的热门类型？
   ├─ 节奏：是否符合快节奏/慢热/中等？
   ├─ 爽点密度：每多少字一个爽点？
   └─ 更新兼容性：适合日更吗？（4000字/章）

2. 留存力评估
   ├─ 读者看完这章，想看下一章吗？
   ├─ 钩子质量：章末钩子是否有效？
   ├─ 主线清晰度：读者知道故事要往哪走吗？
   └─ 期待感：读者在期待什么？

3. 差异化评估
   ├─ 与同类作品相比，有什么独特卖点？
   ├─ 有哪些元素是"别人写过很多次"的？
   └─ 有哪些元素可能成为"出圈"的理由？

4. 平台适配
   ├─ 字数：是否符合平台要求？
   ├─ 敏感内容：是否有审核风险？
   └─ 推荐友好度：前500字是否符合平台推荐算法？
```

### 第5关：一致性审计（V2.1新增 — 数值/战力/知识边界全书级抽样）

> **定位**：前4关聚焦"写得好不好"，本关聚焦"前后对不对"。在全书或大段章节完成后执行，对照00设定的 `power_system` 和 `number_anchors` 做抽样审计。

```
第5关：一致性审计（抽样检查，非逐章）

1. 数值一致性抽样
   ├─ 随机抽取3个角色，核对正文中出现的年龄/修炼枚数/骨骼类型是否与 number_anchors 一致
   ├─ 随机抽取3段距离描写，核对是否与 number_anchors.distances 一致
   ├─ 核对时间线：故事开始至今的时间流逝是否与 number_anchors.timeline 自洽
   └─ 核对物品数量：关键物品的数量/大小是否前后一致

2. 战力一致性抽样
   ├─ 随机抽取3个战斗场景，核对角色表现是否在 power_system 矩阵的 ability_ceiling 内
   ├─ 越级战斗是否有设定依据？（金手指/道具/合击/代价）
   ├─ 旁白描述的危险等级与实际战斗难度是否匹配？
   └─ 战力波动是否有逻辑？（受伤/突破/消耗）

3. 生死状态审计
   ├─ 全书所有"已死亡"角色是否在后续正文中以活人状态出现？（非闪回）
   ├─ 闪回段落是否用了明确的时态/视角切换标记？
   └─ 死亡角色的死亡时间/死因是否与 number_anchors 一致？

4. 知识边界抽样
   ├─ 随机抽取3个信息差场景，核对角色是否说了不该知道的信息
   ├─ 角色对秘密的反应是否有前文信息传递场景支撑？
   └─ 信息差解除的时机是否合理？

输出：一致性审计报告（findings格式，severity blocking/advisory/info）
```

### 多平台评分标准（v2.0 新增）

**审稿时按目标平台加载对应 rubric，无法确定平台时使用通用网文标准：**

| 平台 | 核心评分维度 | 权重侧重 |
|------|------------|---------|
| **番茄小说** | 强开局、强冲突、高频爽点/情绪反馈、低理解门槛 | 爽感密度30%+节奏20%+钩子20%+代入15%+文笔15% |
| **起点中文网** | 设定自洽、升级路径清晰、长线期待、世界观承载力 | 设定30%+结构25%+爽感20%+文笔15%+钩子10% |
| **知乎盐言** | 短篇钩子密集、反转密度高、情绪兑现快、信息差推进 | 反转30%+钩子25%+情绪25%+文笔20% |
| **通用网文** | 综合评估，各维度均衡 | 均等分配 |

**番茄特供审稿要点**（当目标平台为番茄时额外检查）：
1. **黄金三章法则**：前3章必须在3000字内完成"建立人设→制造冲突→给出爽点"三步
2. **爽感密度**：每2000字至少一个情绪反馈点（爽点/笑点/紧张点/泪点）
3. **低理解门槛**：核心设定能否用一句话说清？读者前500字能否理解主角处境？
4. **追读钩子**：章末是否留了"不点下一章就难受"的钩子？连续3章钩子类型不重复
5. **章节定位与张弛**：高压/推进章配齐钩子+爽点+悬念；低压/关系/修炼章允许弱钩子，但每章都给读者一个往下看的理由

## 三档审稿模式（v2.0 新增）

### 模式选择

| 模式 | 触发方式 | 视角数 | 执行方式 | 适用场景 |
|------|---------|:------:|---------|---------|
| **full** | 每10章/卷末自动 + `/review_full` 手动 + 异常（03b连续2章<70分） | 4 | 并行 spawn 4 个 Agent | 卷末/质量滑坡/质量要求高时 |
| **lean** | 每5章自动 + `/review_lean` 手动 + 异常（脚本blocking） + 疲劳防护降级 | 2 | 主线程执行编辑+读者 | 日常章节/快速审稿 |
| **solo**（默认） | `/review_solo` 或无参数手动 | 4 | 主线程串行执行全部 4 关 | 无 subagent 支持时 |

### 降级链

```
full（4 Agent 并行）
  ↓ 条件：环境不支持 subagent / spawn 失败 / 超时
lean（2 Agent 精简）
  ↓ 条件：精简 Agent 也不可用
solo（主线程串行，等同于 V1.0 模式）
```

**每层降级必须报告**：
```
[降级] full → lean：原因 — 当前平台不支持 subagent spawn
[降级] lean → solo：原因 — Agent 配置文件不可读
```

### Full模式Agent协同规范（V2.1补充，V5.1更新Agent引用）

**4 Agent 协同表格**（V5.1：原"读者视角 Agent""商业化评估 Agent"占位符替换为实际 Agent 文件）：

| Agent | 文件 | 视角 | 综合评分权重 |
|-------|------|------|------------|
| story-editor | `.claude/agents/story-editor.md` | 编辑视角（质感） | 30% |
| ai-detector | `.claude/agents/ai-detector.md` | AI鉴定（去AI味） | 25% |
| reader | `.claude/agents/reader-agent.md` | 读者视角（代入感） | 25% |
| commercial | `.claude/agents/commercial-agent.md` | 商业化（卖点） | 20% |

> 第1关（story-editor）复用 `04_小说正文精修师.md` 标准；第2关（ai-detector）复用 `05_去AI味精修师.md` 11 Gate；第3关（reader）与第4关（commercial）为独立 Agent，审查维度见各自文件。

**通信协议**：
- 每个Agent独立运行，产出独立的Findings列表（YAML格式）
- 不需要Agent间实时通信，各自基于同一章节正文独立评审
**Findings合并机制**：
1. 收集4个Agent的Findings列表
2. 按 severity 排序：blocking(致命)>advisory(建议)>info(提示)
3. 同一位置的多个Agent findings合并为一条，标注来源（如"[编辑+读者]"）
4. 冲突处理：blocking级finding任意Agent提出即标记为"需修改"；advisory/info级按多数票决定
**综合评分公式**：
- 综合得分 = 编辑视角×30% + AI鉴定×25% + 读者视角×25% + 商业化×20%
- 各视角满分100分，综合得分满分100分
- 综合得分≥85为优秀，70-84为合格，<70需返工

### 确定性脚本预检（所有模式共用）

审查前自动运行 4 个确定性脚本，报告作为 baseline 注入各视角：

```bash
node scripts/check-ai-patterns.js --check <正文文件>
node scripts/check-degeneration.js --check <正文文件>
node scripts/normalize-punctuation.js --check <正文文件>
node scripts/check-consistency.js --meta=<JSON元数据> <正文文件>
```

脚本发现 ≥1 blocking 级 finding 时，**先修复脚本发现再进入 LLM 审查**（脚本问题是确定性的，不值得浪费 LLM 上下文）。`check-consistency.js` 的报告直接作为第5关一致性审计的 baseline。

## 统一 Findings Schema（v2.0 新增）

所有视角的输出必须使用统一的 Findings 格式（YAML）：

```yaml
findings:
  - severity: blocking      # blocking=必须修复 / advisory=建议优化 / info=锦上添花
    category: prose        # plot|character|prose|consistency|commercial|consistency_audit  # V2.1新增 consistency_audit
    location: Ch.5 L120-135
    evidence: "他看着远方，心中涌起一股莫名的悲伤"
    issue: "AI味：'看着'+'心中涌起'+'莫名的'三重叠加"
    fix: "改为具体动作：'他蹲下身，捡起一块石头抛了抛'"
    gate: G1               # 对应 Gate 编号（仅 AI 味类）
    source: 编辑视角        # 产出视角名称
```

**severity 等级定义**：
| 等级 | 含义 | 处理要求 |
|------|------|---------|
| blocking | 必须修复 | 不修复不定稿，阻断级问题（原S1/S2合并） |
| advisory | 建议优化 | 可以提升品质（原S3） |
| info | 锦上添花 | 有时间再改（原S4） |
