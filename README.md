## 设计理念

### 核心思想：工业化分工

网文创作的痛点不是"不会写"，而是"写了前面忘了后面、质量波动大、AI 味重"。本 OS 采用**三道工序流水线**：

```
基建层（Step 0-2）  →  生产层（Step 3-4）  →  精装层（Step 5-6）
项目+世界观+宏观结构      细纲+初稿               精修+去AI味
     ↓                       ↓                       ↓
   全书1次             滚动建纲+每章循环            每章循环
```

每一步有明确的**输入/输出/边界/不可越界**。下游发现问题，回溯到源头修复，不向下游打补丁。

### 三大流水线法则

| 法则 | 含义 |
|------|------|
| **衔接包唯一生产者** | 只有 02a（细纲生成）产出衔接包。03a 只读不写。其他步骤严禁修改。 |
| **下游不重复上游工作** | 04 不重做 03a 的基础去 AI 味，05 不重做 04 的潜台词打磨（单一责任人制度，V5.2）。 |
| **回溯而非打补丁** | 精修阶段发现剧情逻辑问题 → 回溯到 03a 扩写甚至 02a 细纲阶段修复，而非在精修时改写情节。 |

### 番茄小说特供优化

- **爽感保护协议**：第 1 章前 500 字必须包含一个爽点
- **感官描写二分法**：日常极简白描 / 核心多维叠加（V4.1 废除全局优先级，不再硬性"触觉/嗅觉 > 听觉 > 视觉"）
- **流氓逻辑约束**：主角话锋犀利、护短、果断，不圣母、不犹豫
- **低理解门槛**：避免复杂设定堆砌，读者 30 秒内能进入剧情

## 快速开始

### 第一次使用？

1. **告诉 AI 你的核心灵感** — 一句话即可
2. **AI 会问你 4 个问题**：类型/平台/字数/偏好
3. **确认后进入 Step 0**：创建项目目录结构
4. **沿着流水线一步步推进**，每步完成后 AI 会等你确认

```
你: "我想写一本都市重生小说，主角回到 2000 年用互联网知识改变命运"
AI: 确认信息 → Step 0 → Step 1 → Step 2 → ... → 出书
```

### 已有部分产物？

如果你已经写了一些内容，可以**从任意步骤开始**：

| 你已有什么 | 从 Step 几开始 |
|-----------|:---:|
| 只有灵感 | Step 0 |
| 已有设定文档 | Step 2（大纲） |
| 已有大纲 | Step 3（细纲） |
| 已有细纲 | Step 4（扩写） |
| 已有正文初稿 | Step 5（精修） |
| 已在平台发了部分章节 | Step 扩展 — 07(导入师) |

> AI 会要求你提供已完成的产物，确保下游质量不受影响。

### 模式控制

在任意 Skill 的 Prompt 开头添加模式指令。**两组正交可组合**（V5.1）：

**行为模式**（控制 AI 自主度）：

| 指令 | 效果 |
|------|------|
| `/mode_assisted`（默认） | 全规则静默运行，减少主动建议 |
| `/mode_full` | 全规则激活，主动给出建议和提醒 |
| `/mode_hardcore` | 仅核心约束，跳过主动干预模块 |

**写作流程**（控制流水线步骤数，V5.1）：

| 指令 | 流程 | 适用场景 |
|------|------|---------|
| `/flow_full`（默认） | 03a → 03b → 04 → 05 → 脚本 | 完整4步，质量优先 |
| `/flow_fast` | 03a → 03b → 04b → 脚本 | 快速2-3步，03b≥80分时进入 |
| `/flow_raw` | 03a → 03b → 05精简版 → 脚本 | 生肉2步，快速产出 |

> 两组正交可组合，如 `/mode_assisted /flow_fast`。03b 为所有流程的必经环节（V5.2）。

## 七步流水线详解

### Step 0 — 项目初始化师

| 属性 | 值 |
|------|-----|
| **文件** | `00.5_项目初始化.md` (V1.0) |
| **频率** | 全书 1 次 |
| **输入** | 书名、作者、类型、平台、目标字数 |
| **产出** | 标准目录结构 + `上下文管理模板.md` |

**干什么**：创建 `设定/` `大纲/` `细纲/` `正文/` `追踪/` 五个标准目录，初始化断点续跑追踪文件。所有后续步骤的产出物都落到这个目录结构里。

**目录结构**：
```
{书名}/
├── 设定/          ← Step 1 设定文档 + JSON 元数据
├── 大纲/          ← Step 2 分卷大纲 + 情绪曲线图
├── 细纲/          ← Step 3 全书细纲 + 逐章衔接包链
│   └── 衔接包链/  ←  每章一个独立衔接包文件
├── 正文/
│   ├── 初稿/      ← Step 4a 产出（逐章）
│   ├── 精修/      ← Step 5 产出（逐章）
│   └── 终稿/      ← Step 6 产出（逐章）
├── 追踪/          ← 上下文管理模板 + 伏笔追踪 + 断点快照
└── 对标/          ← 可选（Step 06 拆文产出）
```

### Step 1 — 小说设定架构师

| 属性 | 值 |
|------|-----|
| **文件** | `00_小说设定架构师.md` (V2.6) |
| **频率** | 全书 1 次 |
| **输入** | 核心灵感 + 基本要求 |
| **产出** | 结构化设定文档 + JSON 元数据 |

**产出的 JSON 元数据是整条流水线的标准数据接口**，包含：

```json
{
  "novel_meta": {     // 书名/类型/平台/字数/核心标签
  },
  "characters": [],    // 角色名(唯一标识)/性格/成长弧/关系
  "relationships": [], // 角色关系矩阵
  "anti_patterns": {   // 反例对照表 — 下游所有 skill 加载为约束
    "forbidden": [],   //   禁止使用的情感词/模板句式
    "required": []     //   必须遵守的约束（如感官二分法）
  }
}
```

**下游消费者**：Step 2（大纲）、Step 3a/3b（细纲）、Step 4a/4b（扩写/质检）、Step 5（精修）、Step 6（去 AI 味）——全部依赖此 JSON。

**核心能力**：
- 番茄平台合规审查（敏感元素预检）
- 金手指设计 + 成长阶梯（升级体系）
- 角色关系动力学（CP/对手/师徒等）
- 素材融合与设定保鲜机制
- 防御性设计（扩展性与一致性自检）
- 题材模块选择器（6 题材 × 模块矩阵，非玄幻题材跳过战力矩阵）

### Step 2 — 小说大纲构建师

| 属性 | 值 |
|------|-----|
| **文件** | `01_小说大纲构建师.md` (V2.5) |
| **频率** | 全书 1 次 |
| **输入** | Step 1 的设定文档 + JSON 元数据 |
| **产出** | 分卷大纲 + 情绪曲线 + 核心卖点定位 + INC 13 要素 |

**两阶段设计**：

**第一部分 — 读者方向诊断**：在情节设计之前，先回答"读者到底在为什么买单"——爽感/恐惧/情感拉扯/智商碾压/成长逆袭。

**第二部分 — 结构设计工具箱**：
- **角色位抽象**：将角色抽象为功能位（主角位/反派位/导师位/CP 位/谐星位/工具人位等），确保每个位都有不可替代的剧情功能
- **钩子 20 式**：章末 13 式 + 章首 7 式，连续 3 章不重复
- **反转 7 种类型**：身份/立场/真相/强弱/情感/认知/命运，含节奏控制
- **INC 13 要素**（V5.1）：每卷 ≥6 项（终卷 ≥8 项），大纲产出后必检
- **拆文推荐框架**：如何利用拆文数据指导大纲设计

**产出包含**：
- 每卷核心冲突与爽点分布图
- 黄金三章规划（前 3 章每一章的钩子/情绪温度/爽点类型）
- 全书伏笔计划
- 情感温度曲线预标注

### Step 3 — 细纲生成 + 质检（02a / 02b）

V5.2 将原 `02_细纲编写技能.md`（存档）拆分为两个独立文件，将"生成"与"质检"分离到不同上下文。

#### 02a — 细纲生成

| 属性 | 值 |
|------|-----|
| **文件** | `02a_细纲生成.md` (V5.2.0) |
| **频率** | 滚动建纲（默认）/ 一次性全书建纲（/mode_full） |
| **输入** | Step 1 JSON 元数据 + Step 2 分卷大纲 |
| **产出** | 逐章细纲 + 衔接包链 + 伏笔追踪表 |

**职责**：只负责"生成"——将 01 分卷大纲拆解为逐章细纲。衔接包是 02a 的唯一生产物，03a 扩写系统只能读取。

**章节蓝图五段式**（V4.3 起）：
```
起因 → 发展 → 转折 → 高潮 → 结尾
```

**字数预算编排**：密(≥250字) / 疏(≈40字) / 铺垫(120-150字)

**衔接包 P0/P1 字段分级**（V5.2）：P0 字段（禁止矛盾清单/角色状态）必填且强校验，P1 字段（延续细节/伏笔追踪）按需填写。

**滚动建纲模式**（默认）：
1. 首次产出前 10 章细纲 + 衔接包链
2. 进入 03a 逐章扩写
3. 每写完 3 章触发"补纲"——回到 02a 补充后续 5-10 章细纲
4. 已进入写作的细纲**锁定**，未经确认不得修改

> 全书 ≤30 章时可一次全建。使用 `/mode_full` 可强制全书建纲。

#### 02b — 细纲质检

| 属性 | 值 |
|------|-----|
| **文件** | `02b_细纲质检.md` (V5.2.0) |
| **频率** | 每次 02a 产出后 / 补纲后 |
| **输入** | 02a 产出的细纲 + 衔接包链 |
| **产出** | 质检报告 + 7 项质量 Gate 结果 |

**职责**：独立于 02a 运行，只负责"检测"。包含大纲五检（Part O5：每卷 5 问 + 每章 6 问，含用户情绪锚点）、细纲质量 Gate Part O6（7 项全过才放行）、衔接包自检（11 项，P0/P1 分级校验）、题材→钩子映射表（Part O7）。

**衔接包内容**（每章一个）：
- 前章回顾 + 角色状态表
- 延续细节（需要在本章出现的上章设定）
- **禁止矛盾清单**（如"李四左臂骨折"→本章不得双手操作）
- 新增状态（本章新增的状态/道具/关系变化）
- 伏笔追踪（ACTIVE → DORMANT → WAKING → CLOSED）

### Step 4a — 扩写执行（03a）

| 属性 | 值 |
|------|-----|
| **文件** | `03a_扩写执行.md` (V5.2.0) |
| **频率** | **逐章循环** |
| **输入** | 每章细纲 + 衔接包 + Step 1 反例对照表 |
| **产出** | 初稿（3500-4500字） + P0 预检 + 19 项终检报告 |

**核心使命**：骨架血肉——剧情推进流畅、逻辑不崩、字数达标、基础去 AI 味。**不做深度润色**（那是 Step 5 的工作）。

**规则分层体系**：
- **铁律**（违反 = 重写）：字数/章节结构/对话比例/衔接包禁止矛盾
- **弹性建议**（可破但要知道为什么破）：句式多样性/场景节奏
- **风格可选项**（按题材适配）：打斗密度/甜度/虐度

**场景驱动加载路由器**（V5.0）：按场景类型动态加载 references，核心常驻集 5 个 + 场景扩展集每章最多 3 个，上下文负担降低 60-70%。

**交付给 03b**：初稿 + P0 预检结果 + 终检报告。

### Step 4b — 质量自检（03b）

| 属性 | 值 |
|------|-----|
| **文件** | `03b_质量自检.md` (V5.2.0) |
| **频率** | **每章必执行，不可跳过**（V5.2 强制） |
| **输入** | 03a 产出的初稿 + 终检报告 |
| **产出** | 质检报告 + 5 维质量 Gate 评分 + 流程路由建议 |

**核心使命**：独立质检——在**独立上下文**中执行（禁止与 03a 同一会话），确保检测结果不受生成过程"确认偏误"影响。

**5 维质量 Gate 评分**（V5.2 更新）：
- 情节推进 25 + 情绪到达 25 + 角色一致 20 + 质感密度 15 + 节奏控制 15 = 100
- **≥80 分**：走快速通道（`/flow_fast` → 04b 深度打磨）
- **60-79 分**：走完整流程（`/flow_full` → 04 精修 + 05 去AI味）
- **<60 分**：整体重写（回到 03a）

**P0/P1/P2 分级**：P0 在 03a（5 项必查），P1/P2 在 03b（8+9 项），差异化执行确保可靠性。

### Step 5 — 小说正文精修师（04）

| 属性 | 值 |
|------|-----|
| **文件** | `04_小说正文精修师.md` (V5.2.0) |
| **频率** | **逐章循环**（`/flow_full` 流程） |
| **输入** | 03b 质检通过的初稿 + 质检报告 |
| **产出** | 精修正文（潜台词/金句/风格注入） |

**核心使命**：灵魂质感——不改变情节、不改变设定、不改变结构，只让读者读完"身体有反应"。

**第零部分：网文顶级质感范式**（V5.2.0 调整）：

从"叙述者视角"转换为"沉浸式画面感视角"，审美基准改为猫腻的文人气韵、烽火戏诸侯的江湖质感、愤怒的香蕉的细腻日常、烟雨江南的冷峻氛围。

**两大核心原则 + 冰山原则降级**（V5.2.0 从五大铁律精简）：
1. **潜台词拦截器**（第一原则）：禁止直接写情绪词（"他很生气"→写他捏碎了杯子）
2. **比喻质量门**（与 03a 对齐）：低质量比喻替换为物理痕迹法/生理反馈法，高质量比喻保留；比喻密度由 03a 管控
3. **冰山原则**（降级为高情绪场景可选技法）：网文需要适度情绪直给，完全禁止情绪词会让高情绪场景冷漠疏离
- ~~防顺畅错位引擎~~（V5.2.0 删除：过度追求错位会让文本不自然）
- ~~道具去戏剧化白描~~（V5.2.0 删除：与 Show Don't Tell 重复，且过于文学化）

**五大深度技法**（V5.2.0 从六大精简，删除反标签行为法）：
- 感官入口重定向（与 V4.1 二分法对齐：日常极简白描/核心多维叠加）
- 脏感注入（灰尘/锈迹/油腻/潮湿/磨损）
- 潜台词对话法（试探→回避→松动）
- 内心矛盾外化法（言行不一呈现冲突）
- 感官锚点深度穿插（跟着情绪走，不跟着数字走）
- 标点规范化（省略号 ≤5/破折号 ≤4/感叹号 ≤15 每章）
- 闲笔与呼吸感（V5.2.0 降级为建议：过渡章可选，高潮章不建议）

**职责边界声明**（V5.2.0 新增）：与 03a/03b/05 的单一责任人制度对齐，不重复检测上游已处理项。

### Step 6 — 去 AI 味精修师（05）

| 属性 | 值 |
|------|-----|
| **文件** | `05_去AI味精修师.md` (V1.7) |
| **频率** | **逐章循环**（`/flow_full` 流程） |
| **输入** | Step 5 精修正文 |
| **产出** | 终稿 + AI 味评分报告 + 脚本检查报告 |

**核心使命**：人味儿质检——让文字读起来像真人写的，而非 AI 生成的。

**11 Gate 系统**（顺序检测，不可跳跃）：

| Gate | 名称 | 检测内容 | 级别 |
|:----:|------|---------|:----:|
| 1 | 禁用词过滤 | 一级/二级禁用词清单 | 阻断 |
| 2 | 句式去套路 | AI 固定句式模板 | 阻断 |
| 3 | 心理描写外化 | 内部心理→外部行为 | 核心 |
| 4 | 节奏打碎 | 打破 AI 匀速推进 | 核心 |
| 5 | 对话去腔调 | 消除教科书腔 | 核心 |
| 6 | 结尾去升华 | 消除排比/哲理/情感归纳 | 阻断 |
| 7 | 意象疲劳检测 | 情绪容器反复使用 | 警告 |

| 11 | 解释腔/上帝感/安排感 | 作者意志暴露 | 核心 |

**工程化保护体系**（V1.3 新增）：
- **白名单豁免**：`.deslop-whitelist` 文件中声明的句式/词汇不检测

## 工程化支撑体系

### 确定性脚本 (`scripts/`)

9 个 Node.js 脚本 + 1 个共享模块提供 LLM 无法自检的确定性校验。**只报告不修改**——LLM 读取报告后自行修正。公共函数已提取到 `lib/prose-utils.js`（V5.2），角色名精确匹配避免误报。

| 脚本 | 检测项 | 最高级别 |
|------|------|:---:|
| `check-ai-patterns.js` | 否定翻转("不是A而是B")/破折号(——)/碎句号/长段落/burstiness 突发度 | **阻断** |
| `check-degeneration.js` | 逐字复读/截断/占位符泄漏/工程词泄漏/章节结构词泄漏 | **阻断** |
| `normalize-punctuation.js` | 省略号/破折号残留/markdown 分隔线/引号风格 | 建议 |
| `check-consistency.js` | 数值一致性/战力链/角色名精确匹配（配置驱动，从 metadata 读取题材） | **阻断** |
| `check-rhythm.js` | 奖励间隔/节奏崩塌/情绪平坦/比例失衡/缺失钩子/句长方差 | 警告 |
| `extract-used-patterns.js` | 提取已用句式清单供下章喂入闭环（V5.1） | 信息 |
| `compress-handoff.js` | 超 10 章旧衔接包自动压缩为摘要（V5.1） | 信息 |
| `sync-versions.js` | 跨文件版本号同步检查（V5.1） | 信息 |
| `check-emotion-density.js` | 情绪锚点密度/情绪词残留/情绪反差（V5.2 降级为 advisory） | advisory |
| `lib/prose-utils.js` | **共享模块**：公共函数库（V5.2 新增，避免重复实现） | — |

**使用方式**（Step 6 完成后 / 03b 质检时）：
```bash
node scripts/check-ai-patterns.js --fail-on=blocking 正文/终稿/Ch1_终稿.md
node scripts/check-degeneration.js --fail-on=blocking 正文/终稿/Ch1_终稿.md
node scripts/normalize-punctuation.js --check 正文/终稿/Ch1_终稿.md
node scripts/check-emotion-density.js --anchor "用户情绪锚点" 正文/终稿/Ch1_终稿.md
```

- `--check`：只检查不修改（normalize-punctuation 默认会修改，加此参数仅报告）
- `--json`：输出 JSON 格式（供 LLM 解析）
- `--fail-on=blocking`：遇到阻断级 finding 时 exit 1
- `--anchor`：check-emotion-density 专用，传入 02 细纲的用户情绪锚点验证交付

## 跨平台部署

### 平台配置对照

| 平台 | 配置目录 | Skill 安装位置 | Agent 部署 | Hook 格式 |
|------|---------|--------------|-----------|-----------|
| **OpenCode**（默认） | `.opencode/` | `~/.config/opencode/skills/` | 主线程降级 | `.ps1` / `.sh` |
| **Claude Code** | `.claude/` | `.claude/skills/` | `.claude/agents/`（5个） | `.sh`（bash） |
| **Codex** | `.codex/` | `.codex/skills/` | 主线程降级 | `.sh`（bash） |
| **Trae** | `.trae/` | `.trae/skills/` | `.trae/agents/`（5个，与.claude/对等） | `.ps1` / `.sh` |

### 部署原则

1. 所有 `.md` skill 文件是平台无关的纯 Prompt，任何 AI IDE 都可直接加载
2. Hooks 按平台格式提供：Windows 用 `.ps1`，Linux/Mac 用 `.sh`
3. Agent 配置文件双部署到 `.claude/agents/` 和 `.trae/agents/`，不可用则降级为主线程执行
4. Settings 文件不跨平台覆盖——部署时只写当前平台的配置
