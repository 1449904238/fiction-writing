## 概述

此技能是小说创作的总控流程，串联**7步核心流水线 + 4个扩展技能 + 3个确定性脚本**，按固定顺序逐阶段推进。每个阶段完成后必须等待用户确认，不可自行跨越。

## 7步流水线

| 步骤 | 子技能 | 执行频率 | 产物 |
|------|--------|:---:|------|
| 第0步 | `项目初始化师` | 全书1次 | 标准目录结构 + 上下文管理模板 |
| 第1步 | `小说设定架构师` | 全书1次 | 世界观、人物设定、成长体系、关系动力学等结构化设定文档 |
| 第2步 | `小说大纲构建师` | 全书1次 | 完整情节大纲，符合签约标准 |
| 第3步 | `小说细纲编写技能` | 滚动建纲 | 章节级细纲（五段式+字数预算+大纲五检），先建前10章，每写3章补5-10章 |
| 第4a步 | `细纲扩写执行` (03a) | **逐章循环** | 小说正文初稿（生成+质感注入+情绪工程） |
| 第4b步 | `质量自检` (03b) | **逐章循环** | 质检报告（P1抽查+P2可选+5维评分+快速通道判断） |
| 第5步 | `小说正文精修师` (04) / `深度打磨` (04b) | **逐章循环** | 精修后的正文（潜台词、金句、风格） |
| 第6步 | `去AI味精修师` (05) | **逐章循环** | 最终定稿（11 Gate + 工程化保护 + 确定性脚本兜底） |

> 第0-3步 = 一次性准备（第3步支持滚动建纲，全书≤30章可一次全建） | 第4a-6步 = 逐章循环（每章重复）
> V5.2: 第4b步（03b质检）为**每章必执行**，在独立上下文中运行，质检评分决定后续走快速通道（≥80→04b）还是完整流程（60-79→04→05），<60分退回03a重写

## 总控流程规则

### 一、启动阶段

1. 用户提供**核心灵感**和**基本要求**后，先不要直接调用子技能。
2. 先与用户确认以下信息（如用户已提供则跳过）：
   - 小说类型/题材（都市、玄幻、言情、悬疑等）
   - 目标平台（如番茄小说）和预期字数
   - 主角性别与基本定位
   - 有无特别偏好或禁忌（如不要后宫、不要虐主等）
3. **询问用户从哪一步开始**（见下方"指定起始步骤"规则）。
4. 确认完毕后，告知用户流程即将开始，进入对应步骤。

### 二、指定起始步骤

用户可以跳过前面已完成的步骤，直接从任意步骤开始。

**触发条件**：用户明确表示"从第X步开始"、"我已经有大纲了"、"直接写细纲"等。

**处理流程**：
1. 询问用户从哪一步开始（0-6步）。
2. 如果用户选择跳过前面的步骤，要求用户提供跳过步骤的产物（如已有大纲则提供大纲内容）。
3. 如果用户无法提供完整产物，询问是否基于已有内容补全缺失步骤，或强制跳过（可能影响后续质量）。
4. 确认起始步骤和所需材料后，直接从该步骤开始执行。

**示例**：
- 用户："已经有项目结构了，直接从设定开始。" → 确认项目目录 → 从第1步开始
- 用户："我已经有大纲了，直接从细纲开始。" → 要求提供大纲内容 → 从第3步开始
- 用户："设定和大纲都写好了，从第3步开始。" → 要求提供设定和大纲 → 从第3步开始
- 用户："直接从正文开始。" → 要求提供设定、大纲、细纲 → 从第4步开始
- 用户："正文已经写好了，只需要精修。" → 要求提供正文 → 从第5步开始

### 三、每阶段通用规则

**每个阶段（第1-6步）完成后，必须执行以下确认流程：**

1. 子技能输出产物后，向用户展示摘要。
2. 明确询问用户：
   - **"这个阶段的产物是否满意？不满意我可以重新生成或补充。"**
   - **"是否需要压缩上下文？（压缩后之前的详细内容会被精简，但核心设定会保留）"**
   - **"有没有需要补充的内容或修改意见？"**
3. 等待用户明确回复。用户说"可以"、"继续"、"下一步"等确认词后，才能进入下一阶段。
4. 如果用户要求修改，执行修改后重新确认，直到用户满意。

### 四、回退与返工

**本阶段不合格时，必须回退到依赖的上游阶段重新执行。**

| 当前步骤 | 如果产物不合格 | 回退到 |
|---------|-------------|--------|
| 第1步（设定） | 设定文档有根本性问题 | 重新与用户确认核心灵感 |
| 第2步（大纲） | 大纲方向不对/卖点不清晰 | 第1步（重新审查设定） |
| 第3步（细纲） | 细纲结构与大纲冲突 | 第2步（重新审查大纲） |
| 第4步（扩写） | 扩写偏离细纲/衔接包 | 第3步（重写该章细纲） |
| 第5步（精修） | 精修改动破坏了剧情逻辑 | 第4步（重新扩写后再精修） |
| 第6步（去AI味） | 去AI味后改变了文意的 | 退回04产出再处理 |

**原则**：下游发现问题 = 回溯到源头修复——不要在下游打补丁掩盖上游的错误。

### 四.5、正文偏差 → 逆向调整机制（V2.1.1 新增）

> **场景**：写到第N章正文时，发现角色的实际发展走向与设定/大纲/细纲产生偏差，需要微调上游内容以保证后续剧情流畅。
> **与"回退与返工"的区别**：回退是"产物不合格→重做"；逆向调整是"产物合格但走向偏了→主动微调以适配新方向"。这是正向创作调整，不是纠错。

#### 触发条件（满足任一即触发）

1. **角色偏差**：正文中角色的实际行为/性格发展，与00设定文档中的core_tags产生偏移，且偏移方向更优
2. **关系偏差**：正文中两个角色的关系发展，与01大纲中的evolution_path不一致，且新走向更有张力
3. **情节偏差**：正文实际推进的情节线索，偏离了01大纲中该卷的情节规划，但偏离后的情节更自然
4. **伏笔偏差**：正文中无意中制造了一个新的伏笔机会，或关闭了一个原定伏笔的方式与细纲不同
5. **设定偏差**：正文中使用了设定文档中未定义的能力/物品/规则，但该新增丰富了世界观

#### 逆向调整流程（六步法 — 必须按顺序执行）

```
Step 1: 识别偏差
├─ 03扩写或04精修过程中，发现正文走向与上游产物产生偏差
├─ 记录偏差内容：偏差类型/偏差章节/偏差具体描述/偏差原因
└─ 继续完成当前章节的写作（不打断写作流程）

Step 2: 评估影响（当前章节完成后）
├─ 评估偏差影响范围：仅影响当前章？影响后续3-5章？影响整卷？影响全书？
├─ 评估偏差方向：偏差是"变好了"还是"变差了"？
│  ├─ 变好了（更自然/更有张力/更符合角色逻辑）→ 进入Step 3
│  └─ 变差了（逻辑矛盾/人设崩塌/节奏混乱）→ 回退重写当前章，不进入逆向调整

Step 3: ⚠️ 用户确认门控（必须暂停等待用户决定）
├─ 向用户输出偏差报告：
│  ┌─ 偏差类型：[角色偏差/关系偏差/情节偏差/伏笔偏差/设定偏差]
│  ├─ 偏差位置：第N章，具体描写：[引用正文原文]
│  ├─ 原定方向：[设定/大纲/细纲中的原定内容]
│  ├─ 实际走向：[正文中实际发生的内容]
│  ├─ 影响范围：[仅当前章/后续3-5章/整卷/全书]
│  ├─ 建议调整方案：[具体需要修改哪个上游产物的哪些部分]
│  └─ 不调整的后果：[如果不调整，后续会出现什么问题]
├─ 提供三个选项：
│  A. 接受偏差，调整上游产物（推荐）→ 进入Step 4
│  B. 不接受偏差，回退重写当前章 → 退回03重写
│  C. 部分接受（用户自定义折中方案）→ 按用户指示执行
└─ ⚠️ 禁止未经用户确认就修改任何上游产物

Step 4: 执行逆向调整（用户选择A后）
├─ 按影响范围从小到大依次调整：
│  ① 细纲层面调整：修改02中后续章节的细纲+衔接包，使其与新方向一致
│  ② 大纲层面调整（如影响整卷）：修改01中该卷的分卷大纲，更新情节规划
│  ③ 设定层面调整（如影响全书）：修改00中相关角色的core_tags/relationships/anti_patterns
├─ 调整原则：最小改动原则——只改与偏差直接相关的部分，不做连锁大改
├─ 版本管理：在对应文档中标注"[V2 逆向调整 N章] 原XXX→新XXX（原因：正文偏差）"
└─ 同步更新：衔接包链/伏笔追踪表/角色状态表/上下文管理模板

Step 5: 一致性校验
├─ 检查调整后的上游产物与已完成正文是否一致（前N-1章不受影响）
├─ 检查调整后的上游产物与未写章节的细纲是否一致（第N+1章及后续需同步）
├─ 检查伏笔追踪表是否需要更新（如有新伏笔/关闭的伏笔）
└─ 如发现新的不一致 → 回到Step 4继续调整

Step 6: 恢复写作
├─ 调整完成后，继续从第N+1章开始写作
├─ 第N+1章的细纲+衔接包使用调整后的版本
└─ 在项目日志中记录本次逆向调整的完整信息
```

#### 逆向调整的约束

- **频率约束**：同一卷内逆向调整不超过2次。超过2次说明大纲层面有根本性问题，应回退到01重新规划该卷
- **范围约束**：单次逆向调整最多影响1个设定字段+1条关系线+1卷大纲。超出范围 → 回退到01/00重新规划
- **锁定约束**：已完成精修（04）的章节不接受逆向调整（只能向后调整后续内容）
- **记录约束**：每次逆向调整必须在`追踪/项目日志.md`中记录完整信息

### 五、上下文压缩

- 当用户选择压缩上下文时，对已完成阶段的产物进行精简摘要，保留核心关键信息（人物名、关键设定、核心情节节点），删除冗余细节。
- 以精简格式输出压缩后的内容，作为后续阶段的上下文基础。
- 如果用户不压缩，则保留全部内容继续。
- **数值锚点不可压缩**（V2.8新增）：`number_anchors` 和 `power_system` 字段在任何压缩操作中必须完整保留，不得精简。这两个字段是全书数值和战力一致性的唯一基线，一旦在压缩中丢失，后续所有步骤的一致性校验将失效。

### 六、正文扩写特殊规则（第4步）

- 第4步（正文扩写）开始前，额外向用户确认：**"细纲已完成，是否直接开始进行小说正文扩写？"**
- 用户明确同意后才开始扩写。
- 正文扩写可选择"一次性全部生成"或"逐章生成，每章确认"，由用户决定。

### 六.5、三种写作流程（V5.1 新增，V5.2 更新）

> V5.1 将原来的单一流程扩展为三种，用户可在任何章节切换。默认使用 `/flow_full`。
> V5.2 更新：03b质检在三种流程中均为必执行步骤，不可跳过。
> **注意**：`/flow_*` 前缀控制流水线步骤数，与 AGENTS.md 中的 `/mode_*` 前缀（控制AI自主度）正交，可组合使用，如 `/mode_assisted /flow_fast`。

| 流程 | 前缀 | 流程 | 适用场景 | 上下文负担 |
|------|------|------|---------|-----------|
| **完整流程** | `/flow_full`（默认） | 03a→03b(必执行)→04(精修V5.2)→05(去AI味)→脚本收尾 | 重要章节、开篇、卷末、关键转折 | 高（4步全走） |
| **快速流程** | `/flow_fast` | 03a→03b(必执行)→04b(深度打磨)→脚本收尾 | 日常章节、过渡章节、已建立稳定节奏的连续章节 | 中（3步） |
| **生肉流程** | `/flow_raw` | 03a→03b(必执行)→05精简版→脚本收尾 | 初稿快速产出、后续批量精修 | 低（3步，但03b不可省） |

> **V5.2 重要变更**：三种流程中 03b（质检）均为**必执行**步骤，不可跳过。/flow_raw 虽然是最简流程，但03b质检仍必须执行——/flow_raw 的"简"体现在跳过04精修和05完整去AI味，而非跳过质检。

**快速通道准入条件（03b质检报告判断，4项全满足才走快速通道）：**
1. P0 5项必查全部通过
2. P1 8项抽查无S1级问题
3. 5维评分≥80分（**V5.2核心指标：03b质检评分≥80分自动进入快速通道**）
4. burstiness > 0.35（句长方差检测）

> **V5.2 变更**：①quality-checker Agent从准入条件中移除（改为每章必执行，不再作为快速通道的额外门槛）；②"用户审美裁决"从必要条件改为可选步骤（用户可在03a后检查文风，但不阻塞流程）；③准入条件从5项精简为4项，核心判据为03b评分≥80分。
> Agent不可用时降级为主线程自检，不影响快速通道判定逻辑。

**流程切换规则：**
- 用户可随时用前缀切换流程：`/flow_fast 从下一章开始`
- 重要章节（开篇/卷末/高潮章）建议锁定 `/flow_full`，即使用户指定快速流程也应提醒
- `/flow_raw` 产出的章节必须在后续用 `/flow_full` 补全精修+去AI味

**用户审美检查点（V5.1 新增，V5.2 调整为可选）：**
- 03a扩写完成后、进入03b质检前，可向用户展示初稿前200字（可选步骤，不阻塞流程）
- 询问："这个文风和节奏是否符合你的预期？需要调整再继续质检。"
- 用户确认后进入03b；如需调整，修改后重新展示
- **V5.2 变更**：此检查点从"必经环节"降级为"可选环节"——用户可选择跳过直接进入03b质检。快速通道准入条件中不再将"用户审美裁决"作为必要条件

### 七、每3章一致性检查（第4步循环中触发 — V5.2更新）

**扩写完第3、6、9、12...章后（V5.2: 从每5章调整为每3章），暂停扩写，执行以下检查：**

1. **衔接包禁止矛盾链**：逐条核实最近3章的衔接包"禁止矛盾"条目，确认无一违反
2. **伏笔追踪表**：更新所有伏笔状态，确认ACTIVE伏笔有明确回收计划，DORMANT伏笔未触发时限
3. **角色状态漂移**：对照00设定文档，检查角色性格/能力是否与设定一致
4. **字数节奏**：最近3章的字数（4000±500）、对话比（≤60%）、场景数（3-5）是否符合契约
5. **数值与战力校验**（V2.8新增）：①扫描最近3章所有数值（年龄/距离/枚数/境界/物品数量/时间线），对照00设定的 `number_anchors` 字段，发现不一致即标记修正；②检查战斗场景中的角色表现是否与 `power_system` 矩阵的能力边界匹配，越级表现无设定依据即标记修正；③核实已标记为"已死亡"的角色是否以活人状态出现

**检查结果**：
- 全部通过 → 继续扩写
- 发现问题 → 标记问题章节，在扩写下一章时修正（不回溯已完成的章节，除非是P0级矛盾）

**中段加密检查**（V2.8新增，V5.2更新）：全书40%-60%区域（如全书50章则在第20-30章区间），将每3章检查改为每2章检查。实证研究表明AI长篇生成的错误集中在叙事中段，加密检查频率可更早捕获累积偏差。

### 八、精修与去AI味规则（第5-6步）

- 第5步精修：专注于潜台词打磨、金句提炼、个人风格注入。
- 第6步去AI味：去除AI写作痕迹，使文本更自然流畅。
- 两步独立执行，每步完成后都需要用户确认。

### 九、子技能调用方式

每个阶段调用对应的 Skill 工具：

```
第0步: Skill(name="项目初始化师")
第1步: Skill(name="小说设定架构师")
第2步: Skill(name="小说大纲构建师")
第3步: Skill(name="小说细纲编写技能")
第4a步: Skill(name="细纲扩写执行")          # V5.1: 03a_扩写执行.md
第4b步: Skill(name="质量自检")              # V5.1: 03b_质量自检.md
第5步: Skill(name="小说正文精修师")          # 或 Skill(name="深度打磨") 用于/flow_fast
第6步: Skill(name="去AI味精修师")
```

调用后，根据子技能展开的指令执行对应阶段的任务，完成后执行本技能的确认流程。

### 十、中断与恢复

- 用户可随时中断流程。下次继续时，告知用户当前处于哪个阶段，从该阶段继续。
- 如果用户想跳过某个阶段，必须明确确认，不可自行跳过。

### 十一、前置调研入口（v2.0 新增）

**在第2步（大纲）之前，建议先调用 06_小说拆文师 执行前置市场调研：**

1. 如果用户是新书项目（从第0步或第1步开始），在第2步前主动询问：**"是否需要先做前置市场调研（扫榜+拆文对标）？可以帮助确定题材方向和卖点。"**
2. 用户确认后，调用 `06_小说拆文师` 执行：扫榜（Phase 1-4）→ 拆文 → 对标数据写入 `对标/` 目录
3. 02 编写细纲时读取 `对标/` 数据，把对标书角色当功能位，反哺大纲设计

### 十二、滚动建纲触发（v2.0 新增，V5.2更新）

**第3步（细纲）默认采用滚动建纲模式：**

1. 首次产出前 10 章细纲 + 衔接包链（全书≤30章时可一次全建）
2. 进入第4步逐章扩写，每写完 3 章触发"补纲"：回到第3步补充后续 5-10 章细纲（V5.2: 从每5章调整为每3章）
3. 已进入正文写作的前 10 章细纲**锁定**，未经用户确认不得修改
4. 用户可指定"一次性全书建纲"替代滚动建纲

### 十三、确定性脚本收尾（v2.0 新增，V5.1扩展）

**第6步（去AI味）完成后，自动执行确定性脚本兜底检查：**

```
node scripts/check-ai-patterns.js --fail-on=blocking 正文文件
node scripts/check-degeneration.js --fail-on=blocking 正文文件
node scripts/normalize-punctuation.js --check 正文文件
node scripts/check-consistency.js --meta=设定/JSON元数据.json 正文文件
```

- `check-ai-patterns.js`：检测否定翻转/破折号/碎句号/长段落 → blocking 必须清零
- `check-degeneration.js`：检测逐字复读/截断/占位符/工程词泄漏 → blocking 必须清零
- `normalize-punctuation.js`：清理残留省略号/破折号 → --check 模式只报告
- `check-consistency.js`（V2.8新增）：检测年龄/距离/枚数/生死状态与JSON元数据的一致性 → blocking 必须清零
- 脚本只报告不修改，修改由 LLM 根据报告执行

## 注意事项

1. **不可自行跨越步骤**：每步必须等待用户确认后才能进入下一步。
2. **指定起始步骤时必须验证**：如果用户跳过前面步骤，必须要求提供已有产物，否则质量无法保证。
3. **上下文压缩是可选的**：用户可以选择压缩或不压缩，尊重用户选择。
4. **精修和去AI味是独立步骤**：不要将第5步和第6步合并执行。
5. **第0步为全书奠基步骤**：即使从后续步骤开始，也应确认项目目录结构是否存在。
6. **回退不是惩罚**：下游发现问题回溯到源头修改是正确做法，不要为了"往前走"而掩盖错误。
7. **扩展技能独立可用**：拆文/导入/审稿/短篇可在任何阶段独立调用，不依赖主流水线进度。
8. **前置调研建议但不强制**：新书项目在第2步前建议做前置调研，但用户可跳过。
9. **滚动建纲为默认**：第3步默认滚动建纲，用户可指定一次性全书建纲。
10. **三种写作流程（V5.1新增，V5.2更新）**：`/flow_full`（完整4步）、`/flow_fast`（快速3步）、`/flow_raw`（生肉3步，03b不可省）。重要章节建议锁定完整流程。与 `/mode_*` 行为模式正交可组合。
11. **03a/03b不可跳过**（V5.2强化）：即使使用快速模式或生肉模式，03a扩写和03b质检都必须执行。03b必须在独立上下文中运行（禁止在03a同一会话中继续），质检评分≥80分自动走快速通道（04b），60-79分走完整流程（04→05），<60分退回03a重写。
12. **跨章模式提取（V5.1新增）**：03a扩写前建议运行 `extract-used-patterns.js` 提取近5章句式/意象模式，避免跨章重复。
13. **对抗性质检Agent（V5.1新增，V5.2强化）**：03b质检阶段**每章必调用** `quality-checker` Agent（.claude/agents/）进行独立找茬审查，Agent不可用时降级为主线程自检。V5.2: 从"关键章节时启用"改为"每章必执行"。
14. **确定性脚本是兜底**：脚本只报告不修改，blocking 级 finding 必须清零后才能定稿。
15. **跨平台部署**：skill 文件平台无关，hooks/settings 按当前平台约定部署。
16. **每3章审稿触发**（V5.2新增）：09多视角审稿从"每5章"调整为"每3章"（第3/6/9/12...章后触发），与一致性检查同步执行。

## AI使用策略指导（v2.3 新增 — 明确AI能力的边界）

> **核心理念**：AI是"打字员+校对员+结构工程师"，不是"作家"。本流水线的设计哲学是——让AI做AI擅长的事（结构/一致性/量产/检测），让人做人擅长的事（情感判断/创意跳跃/读者共鸣/审美决策）。

### 一、AI擅长的领域（放心交给AI）

| 能力 | 流水线中的体现 | 信任度 |
|------|--------------|--------|
| **结构化输出** | 00设定/01大纲/02细纲——按模板填充结构化字段 | ★★★★★ |
| **一致性维护** | 跨章衔接包/伏笔追踪表/角色状态表——机械记忆不出错 | ★★★★★ |
| **量产能力** | 03扩写——4000字/章的骨架血肉填充 | ★★★★☆ |
| **模式检测** | 05去AI味/check-ai-patterns.js——识别AI味句式/禁用词 | ★★★★★ |
| **信息密度控制** | Gate 10注水熔断——检测"高级废话" | ★★★★☆ |
| **格式规范** | 标点/字数/对话比——机械约束执行 | ★★★★★ |

### 二、AI不擅长的领域（必须人工介入）

| 能力 | 为什么AI做不好 | 流水线的应对 |
|------|-------------|-------------|
| **情感判断** | AI没有情感体验，只能"模拟"情绪表达 | 情绪工程协议——不要求AI"有情感"，而是让AI"设计能触发读者情感的结构"（情绪蓝图→情绪锚点→情绪通路） |
| **创意跳跃** | AI倾向于"最可能的下一个"，而非"最惊喜的下一个" | 受控失控/AI创造配额——每章允许1处偏差，但必须登记；重大创意决策由用户确认 |
| **读者共鸣** | AI不知道"什么让读者爽"——它只知道"什么让文本通顺" | 爽点三段式+情绪公式检验——用结构化公式约束"爽感必须到达" |
| **审美决策** | "这句好不好""这个比喻妙不妙"——AI的审美判断力弱 | 04精修师+用户确认门控——AI产出候选，人类做最终审美裁决 |
| **风格个性** | AI默认"安全正确"，缺乏个人辨识度 | 角色语言指纹/遮名测试——用机械方法逼AI写出有差异的声音 |

### 三、何时信任AI vs 何时覆盖

**信任AI的情况**：
- 结构化字段填充（设定/大纲/细纲的模板字段）→ AI填完后快速浏览即可
- 一致性检查（衔接包/伏笔/角色状态）→ AI的记忆力远超人类
- AI味检测（禁用词/句式套路/比喻密度）→ 信任脚本的机械检测
- 字数/标点/对话比等量化约束 → 信任AI的计数能力

**必须覆盖AI的情况**：
- **情绪弧线设计**：AI的情绪曲线往往是"平缓上升"的安全模式，人类需要手动制造"陡峭的反差"——情绪冲击力 = 预期反差 × 合理度
- **章末钩子选择**：AI倾向于"温和悬念"，人类需要判断"什么钩子让读者忍不住翻下一章"
- **角色对话个性**：AI的对话默认"礼貌得体"，人类需要注入"角色特有的粗糙感"
- **"删还是留"的判断**：AI倾向于"保留所有信息"（怕丢信息），人类需要判断"这段读者真的需要吗"
- **伏笔回收时机**：AI倾向于"尽快回收"（怕忘），人类需要判断"什么时候回收最有冲击力"

### 四、流水线的AI弱点对冲设计

本流水线通过以下机制系统性对冲AI的固有弱点：

```
AI弱点：没有情感 → 对冲：情绪工程协议（设计情感结构而非模拟情感）
AI弱点：安全正确 → 对冲：受控失控配额（强制留出创造空间）
AI弱点：句式均匀 → 对冲：突发度检测 + 三短一长交替要求
AI弱点：解释本能 → 对冲：Gate 11解释腔检测 + 冰山原则
AI弱点：升华本能 → 对冲：Gate 6结尾去升华 + 斩断解释性后摇
AI弱点：信息注水 → 对冲：Gate 10注水熔断 + 反注水最高铁律
AI弱点：比喻泛滥 → 对冲：比喻质量门 + 比喻密度检测
AI弱点：重复劳动 → 对冲：交接协议 + 职责去重机制
AI弱点：改了没改对 → 对冲：修复效果验证闭环
```

### 五、用户在流水线中的角色

用户不是"等待AI产出然后点头"的旁观者，而是以下四个关键角色：

1. **创意总监**：在00-02阶段做核心创意决策（题材/卖点/角色/情节走向）
2. **审美裁决者**：在04精修阶段做"好不好看"的最终判断
3. **情感校准器**：在03扩写后检查"这段是否打动了你"——AI不知道，但你知道
4. **质量守门人**：在05去AI味后做最终通读——"读起来像人写的吗"

> **一句话**：AI负责"写出来"，你负责"判断对不对"。流水线把AI的产出切成了足够小的片段（逐章/逐Gate），让你能在每个关键节点介入，而不是面对一整本书的AI文本无从下手。
