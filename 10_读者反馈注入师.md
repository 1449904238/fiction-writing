# 读者反馈注入师（V2.0 — 读者反馈闭环）

> 本技能为可选步骤，在09多视角审稿师之后执行（按卷触发）。
> 核心理念：将番茄平台的读者行为数据回流入创作流程，驱动后续章节的节奏/写法调整。
> **V2.0 重大更新**：① 完读率诊断方法论（三级分类+断崖检测）；② 评论分析方法论（高评论/吐槽/催更分类处理）；③ 趋势分析方法论（收藏/推荐票趋势）；④ 自动化方案（WebSearch获取公开评论+三级降级链）；⑤ Findings Schema输出（与09对齐）；⑥ 02a逆向调整机制联动；⑦ 用户情绪锚点验证。

## 适用场景
- 每完成一卷后执行（约30-50章）
- 番茄后台数据更新后按需执行
- 完读率异常波动时紧急触发
- 读者评论出现集中吐槽/催更时触发

## 数据获取 — 三级降级链（V2.0新增）

> 参照06拆文师的三级降级链设计，读者反馈数据获取同样采用"手动→WebSearch→定性推测"三级降级策略，确保任何情况下都能产出分析报告。

### Level 1（优先）：手动输入 — 用户提供番茄后台数据

用户提供以下数据（手动粘贴/截图描述）：
- 番茄完读率数据（按章节）
- 章节评论摘选（高赞评论/差评评论）
- 收藏/推荐数据变化趋势
- 弃书节点数据（如有）

- 用户已提供数据 → 直接进入分析方法论，无需降级
- 用户表示"没有数据/不确定" → 降级至 Level 2

### Level 2（半自动）：WebSearch — 搜索公开评论与书评

> 使用WebSearch工具自动获取目标作品的公开评论数据。

- 搜索关键词示例：`"书名" 番茄小说 评论` / `"书名" 读后感` / `"书名" 吐槽`
- 从搜索结果中提取：读者评价关键词、常见吐槽点、好评亮点
- 搜索结果存入 `对标/读者反馈_WebSearch.md`
- **限制说明**：WebSearch无法获取番茄后台的完读率/收藏/推荐票等私有数据，仅能获取公开评论
- 搜索无结果/数据过少 → 降级至 Level 3

### Level 3（兜底）：定性推测 — 基于已有正文的定性分析

> 当外部数据完全不可获取时，基于已有正文和02a细纲进行定性推测分析。

- 分析已有正文的节奏/钩子/情绪密度，推测可能的读者反应
- 参照09审稿师的读者视角评分，推测完读率趋势
- 明确告知用户："当前为定性推测模式，建议手动核实番茄后台数据"
- 输出"数据置信度"标记：高（后台数据）/中（公开评论）/低（定性推测）

### 降级触发条件汇总

| 异常类型 | 降级动作 |
|---------|---------|
| 用户已提供完整后台数据 | 直接进入分析，不降级 |
| 用户表示无数据/拒绝提供 | → Level 2 |
| WebSearch无结果/数据过少 | → Level 3 |
| 用户仅提供部分数据（如有评论无完读率） | 混合模式：已有数据用Level 1分析，缺失部分用Level 2/3补充 |

### 私有数据获取替代方案（V6.0.1新增 — 补全三级降级链的执行细节）

> **痛点对症**：番茄后台的完读率/收藏/推荐票等核心数据为私有数据，WebSearch无法获取。原三级降级链仅说明"降级至Level 2/3"，但未提供Level 1的人工输入模板、Level 2的WebSearch关键词优化、Level 3的定性推测方法论，导致LLM在数据获取环节卡壳。本节补全三个Level的具体执行方案。

  Ch.X: XX%
  Ch.X: XX%
  Ch.X: XX%
*十万字完读率：XX%
卷均完读率：XX%（如知道）

流量数据：
追读率：XX%（如知道）

章评数据（至少提供3条高赞评论）：
- Ch.X 高赞评论："..." (XX赞)

> - **书架比**：番茄后台"数据总览"页可查看；反映收藏转化
> - **催更人数**：评论区"催更"标签下可统计；反映期待度
> - **十万字完读率**：番茄后台核心留存指标；平台判断书籍潜力的关键数据

#### Level 2增强：WebSearch关键词优化建议

> WebSearch无法获取番茄后台私有数据，但可获取公开评论。以下关键词组合可提升搜索效率。

**【WebSearch关键词组合表】**

| 搜索目标 | 关键词组合 | 预期获取数据 | 数据质量 |
|---------|-----------|-------------|---------|
| 读者评论 | `番茄小说 [书名] 读者评论` | 章评摘选/读者态度 | 中 |
| 吐槽/差评 | `[书名] 吐槽` / `[书名] 差评` / `[书名] 弃书` | 负面反馈集中点 | 中 |
| 读后感 | `[书名] 读后感` / `[书名] 书评` | 深度评价/优缺点分析 | 中高 |
| 社交媒体讨论 | `[书名] 知乎` / `[书名] 贴吧` / `[书名] 微博` | 跨平台读者反馈 | 中 |
| 数据排行 | `番茄小说 [书名] 排行` / `[书名] 在读数` | 榜单排名/在读数（可能过时） | 低 |

> **搜索策略建议**：①先搜`[书名] 吐槽`获取负面反馈（最 actionable），再搜`[书名] 读后感`获取深度评价；②跨平台搜索（知乎/贴吧/微博）可交叉验证读者态度；③搜索结果存入 `对标/读者反馈_WebSearch.md`，标注采集日期。

#### Level 3增强：定性推测方法论

> 当外部数据完全不可获取时，基于已有正文和02a细纲进行定性推测。以下方法论确保推测有据可依，而非凭空臆断。

**【定性推测方法论 — 从正文特征推断读者反应】**

| 推测维度 | 正文特征指标 | 推断读者反应 | 完读率趋势预判 |
|---------|-------------|-------------|---------------|
| 节奏密度 | 核心场景占比<25%或过渡场景>40% | 节奏拖沓，读者可能跳读 | 偏低 |
| 钩子有效性 | 连续3章使用同一类型钩子 | 钩子失效，读者可能弃书 | 偏低 |
| 情绪锚点交付 | 02a设定情绪锚点但正文未交付 | 读者期待落空 | 偏低 |
| 信息密度 | INC命中<3项/章 | 信息含金量低，读者觉得"水" | 偏低 |
| 角色行为一致性 | 角色行为与00设定矛盾 | 读者出戏/OOC吐槽 | 偏低 |
| 爽点频率 | 连续3章无爽点 | 读者疲劳 | 偏低 |
| 高光章节 | 反转+爽点+情绪爆发三合一 | 读者推荐/催更 | 偏高 |

**【定性推测执行流程】**
1. 从02a细纲提取每章的场景比例/钩子类型/情绪锚点/INC命中数
2. 按上表逐章评估正文特征指标
3. 标记"偏低"指标集中的章节为"推测问题章节"
4. 参照09审稿师的读者视角评分，交叉验证推测结果
5. 输出"定性推测报告"，明确标注"当前为定性推测模式，建议手动核实番茄后台数据"

#### 数据质量分级标注（V6.0.1新增）

> 所有读者反馈数据须标注数据质量分级，确保下游02a调整建议的置信度可追溯。

| 数据来源 | 数据质量分级 | 标识 | 可支撑的调整决策 |
|---------|------------|------|----------------|
| 一手后台数据（完读率/追读率/书架比） | 高 | `data_confidence: 高` | S1级Findings（必须修复） |
| WebSearch公开评论 | 中 | `data_confidence: 中` | S2/S3级Findings（建议调整） |
| 定性推测（基于正文特征推断） | 低 | `data_confidence: 低` | S3/S4级Findings（参考优化） |

> **分级使用原则**：①高置信度数据可直接触发02a逆向调整；②中置信度数据须交叉验证2个以上来源后才触发调整；③低置信度数据仅作为参考，须明确告知用户"建议手动核实后再调整"。

## 番茄读者反馈标准化模板（V5.2新增）

> 将番茄平台读者行为数据标准化为统一 YAML 格式，便于自动分析与跨卷对比。本模板定义数据采集结构、自动分析规则与 Gate 触发条件，与三级降级链（Level 1/2/3）兼容。

### 标准化数据模板（YAML）

```yaml
# 番茄读者反馈标准化模板 — reader_feedback
collection_date: 2026-07-16          # 数据采集日期
volume: 第X卷                         # 对应卷次
chapters_covered: Ch.1-Ch.30          # 覆盖章节范围
data_source: level_1                  # level_1(手动)/level_2(WebSearch)/level_3(定性推测)
data_confidence: 高                   # 高(后台数据)/中(公开评论)/低(定性推测)

metrics:
  chapter_completion_rate:            # 逐章完读率（核心指标，逐章列出）
    - { chapter: Ch.1, rate: 92.3% }
    - { chapter: Ch.2, rate: 88.1% }
    - { chapter: Ch.3, rate: 79.5% }
    # ... 逐章补全
  追读率: 45.2%                        # 追读率（追读人数 / 阅读人数）
  书架比: 12.8%                        # 书架比（加书架人数 / 阅读人数）
  催更人数: 1280                       # 催更人数（累计）
  十万字完读率: 38.6%                  # 十万字完读率（平台核心留存指标）

comments:
  - { chapter: Ch.5, type: 高赞, content: "...", likes: 342 }
  - { chapter: Ch.12, type: 吐槽, content: "...", likes: 89 }
  - { chapter: Ch.18, type: 催更, content: "...", likes: 156 }
```

### 自动分析规则

#### 规则1：完读率跌幅分析

> **触发条件**：某章完读率比前一章下降 > 10%（相对跌幅）即标记为问题章节；下降 > 15% 升级为断崖式下跌。

```yaml
rule: completion_rate_drop
trigger: "chapter_completion_rate[ch].rate - chapter_completion_rate[ch-1].rate < -10%"
action:
  - 标记该章为"问题章节"，记录跌幅值与前章完读率
  - 关联 02a 细纲检查该章内容（场景比例/钩子类型/情绪锚点/INC要素）
  - 跌幅 > 15% → 升级为"断崖式下跌"，触发 S1 级 Findings
output:
  problem_chapters:
    - { chapter: Ch.X, prev_rate: 85%, curr_rate: 72%, drop: 13%, severity: S1 }
```

#### 规则2：追读率趋势分析

> **触发条件**：按卷统计追读率，与上卷对比判断趋势方向。

```yaml
rule: follow_rate_trend
trigger: "metrics.追读率 对比 上卷追读率"
analysis:
  上升_5pct:    正向验证，当前卷节奏/写法有效 → 02a 保持
  波动_5pct内:  中性，节奏正常 → 02a 在后续设计高潮刺激增长
  下降_5pct:    预警，可能进入疲劳期 → 02a 引入新冲突/新角色/新场景
  下降_15pct:   紧急 → 触发完读率<50%紧急诊断流程
output:
  trend: 上升/平稳/下降/紧急
  delta: "+3.2% / -8.5%"
  suggestion: "02a 调整建议"
```

#### 规则3：评论关键词映射 Gate

> **触发条件**：对读者评论提取关键词，映射到质量维度 Gate；命中次数 ≥ 5 触发对应 severity 的 Findings。

```yaml
rule: comment_keyword_gate
keyword_map:
  逻辑硬伤:    { keywords: ["前后矛盾","设定崩了","不合理","bug","穿帮"], gate: consistency, severity: S1 }
  人设崩塌:    { keywords: ["人设崩了","ooc","性格突变","不像XX会做的事"], gate: character, severity: S1 }
  节奏问题:    { keywords: ["拖沓","太慢","水字数","无聊","注水","快进"], gate: rhythm, severity: S2 }
  文笔问题:    { keywords: ["文笔差","ai味","机翻","出戏","不像人写的"], gate: prose, severity: S2 }
  情感共鸣:    { keywords: ["哭了","感动","虐","甜","意难平","破防"], gate: emotion, severity: S4 }
  期待催更:    { keywords: ["催更","快更新","等不及","求加更","熬夜追"], gate: commercial, severity: S4 }
action:
  - 统计每个 gate 的关键词命中次数与出现章节
  - 命中次数 ≥ 5 → 触发对应 severity 的 Findings
  - 正向 gate（emotion/commercial）命中 → 提炼有效写法，注入 02a
output:
  keyword_hits:
    consistency: { count: 8, top_keywords: ["前后矛盾","不合理"], chapters: [Ch.15, Ch.20] }
    rhythm:      { count: 12, top_keywords: ["拖沓","水字数"], chapters: [Ch.8, Ch.22] }
```

### metrics 字段说明

| 字段 | 含义 | 采集方式 | 分析用途 |
|------|------|----------|----------|
| chapter_completion_rate | 逐章完读率 | 番茄后台（逐章） | 完读率跌幅分析 / 断崖检测 |
| 追读率 | 追读人数 / 阅读人数 | 番茄后台（按卷） | 追读率趋势分析 |
| 书架比 | 加书架人数 / 阅读人数 | 番茄后台 | 收藏增长趋势分析 |
| 催更人数 | 累计催更人数 | 番茄后台 / 评论 | 读者期待点识别 |
| 十万字完读率 | 十万字处完读率 | 番茄后台（平台核心指标） | 长线留存评估 |

> **使用方式**：用户按模板填写 YAML 后，本技能自动解析并执行上述 3 条规则，输出 Findings（与 09 Schema 对齐）+ 问题章节标记 + 趋势判断，驱动 02a 逆向调整。

## 分析方法论

### 1. 完读率诊断方法论（V2.0新增）

> 完读率是衡量读者留存的核心指标。按完读率水平分为三级诊断+断崖检测。

#### 1.1 完读率 > 80%：高留存诊断 — 提炼有效写法

**诊断目标**：找出"什么写法有效"，提炼为可复用的写法模板。

| 分析维度 | 提炼内容 | 输出 |
|----------|----------|------|
| 场景结构 | 高完读率章节的场景编排特征（核心/重要/过渡场景比例） | 场景结构模板 |
| 钩子类型 | 章末钩子类型分布（13种章末钩子中哪几种效果最好） | 高效钩子清单 |
| 情绪节奏 | 情绪曲线特征（起伏频率/高低差/核心情绪类型） | 情绪节奏模板 |
| 信息释放 | 信息密度分布（每章释放多少新信息/如何控制信息流） | 信息释放策略 |

**注入方式**：将提炼出的有效写法模板注入02a细纲生成，后续章节优先使用高完读率章节的写法模式。

#### 1.2 完读率 50%-80%：中留存诊断 — 诊断问题

**诊断目标**：找出"哪里出了问题"，标记需调整的章节和维度。

| 问题类型 | 诊断方法 | 判断标准 | 调整建议方向 |
|----------|----------|----------|-------------|
| 节奏拖沓 | 检查低完读率章节的场景数+字数分布 | 过渡场景>40%或核心场景<25% | 02a压缩过渡场景，增加核心场景比重 |
| 角色行为不合理 | 对照00设定角色性格+02a细纲角色行为 | 角色行为与设定矛盾/动机不足 | 02a补充角色动机链 |
| 信息密度过低 | 检查低完读率章节的INC要素命中数 | INC命中<3项 | 02a增加新信息/新冲突/新悬念 |
| 钩子失效 | 检查低完读率章节的章末钩子类型 | 连续3章使用同一类型钩子 | 02a轮换钩子类型 |
| 情绪平淡 | 检查低完读率章节的情绪锚点交付 | 无明确情绪锚点或情绪单一 | 02a强化情绪工程，增加情绪锚点 |

**注入方式**：将问题诊断结果关联到02a调整建议，具体到哪些章节怎么改。

#### 1.3 完读率 < 50%：低留存紧急诊断

**诊断目标**：紧急排查"哪里让读者放弃"，优先修复致命问题。

| 紧急排查维度 | 诊断方法 | 紧急程度 |
|-------------|----------|:--------:|
| 开篇吸引力 | 前3章完读率是否断崖？第1章钩子是否有效？ | P0 |
| 核心冲突 | 主线冲突是否清晰？读者是否能在前5章理解"这本书讲什么"？ | P0 |
| 主角代入感 | 主角是否有明确目标？读者是否能产生共情？ | P0 |
| 信息过载 | 是否在前5章塞入过多设定信息？ | P1 |
| 节奏失衡 | 是否长期无爽点？冲突升级是否过慢？ | P1 |

**紧急处理**：
- P0问题 → 立即回溯02a修改对应章节细纲，必要时回溯01大纲调整核心冲突
- P1问题 → 在后续5章内通过02a调整逐步修正

#### 1.4 断崖式下跌检测

> **断崖定义**：完读率下降>15%（相对前一章或前一批次均值）为断崖式下跌。

**断崖诊断流程**：
1. 定位断崖章节（完读率下降>15%的章节）
2. 对照02a细纲检查该章内容：
   - 是否有重大剧情转折导致读者不适？（如突然虐主/核心角色死亡/画风突变）
   - 是否有逻辑硬伤？（设定矛盾/角色行为不合理/时间线错误）
   - 是否节奏突变？（突然大段设定说明/战斗过长/日常过多）
   - 钩子是否失效？（章末无悬念/悬念过于复杂导致放弃追读）
3. 标记断崖原因，关联到02a调整建议
4. 如断崖原因可修复 → 02a调整后续章节补偿（如断崖因虐主 → 后续增加爽点平衡）
5. 如断崖原因不可逆（如已发布章节） → 02a调整后续节奏避免连续断崖

### 2. 评论分析方法论（V2.0新增）

#### 2.1 高评论章节分析 — 提炼讨论触发点

**诊断目标**：找出"什么引发了讨论"，在后续章节中复制此类写法。

| 讨论触发点类型 | 特征 | 复制策略 |
|---------------|------|----------|
| 争议性话题 | 读者围绕某个道德/价值观议题争论 | 02a设计类似的道德困境/灰色选择 |
| 角色行为 | 读者对角色的某个决定热烈讨论 | 02a增加角色决策场景，强化角色个性 |
| 情节反转 | 读者对反转表示惊讶/讨论可能性 | 02a设计类似幅度的反转（注意不重复） |
| 情感爆发 | 读者被某个情感场景打动 | 02a强化情绪锚点交付，复制有效情绪模式 |
| 悬念设置 | 读者猜测后续发展/讨论可能性 | 02a增加类似悬念，控制信息释放节奏 |

**注入方式**：将高评论章节的讨论触发点类型+写法特征注入02a，后续章节优先使用高讨论度的写法模式。

#### 2.2 读者吐槽处理 — 标记"需修复"项

**处理流程**：
1. 收集读者吐槽（差评/负面评论/弃书评论）
2. 分类标记：

| 吐槽类型 | 标记 | 关联调整 |
|----------|------|----------|
| 逻辑硬伤 | `[需修复-逻辑]` | 关联到02a调整建议，检查设定一致性 |
| 角色人设崩塌 | `[需修复-人设]` | 关联到00设定文档+02a角色行为链 |
| 节奏问题 | `[需修复-节奏]` | 关联到02a场景比例调整 |
| 文笔问题 | `[需修复-文笔]` | 关联到04精修师/05去AI味 |
| 设定矛盾 | `[需修复-设定]` | 关联到00设定文档+number_anchors |

3. 将标记项输出为Findings（与09 Findings Schema对齐，见下方）
4. 关联到02a调整建议，优先处理标记最多的吐槽类型

#### 2.3 读者催更分析 — 识别期待点

**处理流程**：
1. 收集读者催更评论（"快更新""等不及了""求加更"）
2. 分析催更关注点：
   - 读者在催什么线索的推进？（→ 该线索的期待值高）
   - 读者在催什么角色的发展？（→ 该角色的人气高）
   - 读者在催什么悬念的揭晓？（→ 该悬念的钩子力强）
3. **提前回收策略**：
   - 高期待线索 → 02a提前该线索的推进/回收时间（原计划第3卷回收 → 提前到第2卷）
   - 高人气角色 → 02a增加该角色戏份
   - 高钩子力悬念 → 02a控制释放节奏，不过早揭晓也不过晚回收

### 3. 趋势分析方法论（V2.0新增）

#### 3.1 收藏增长趋势分析

| 趋势 | 判断 | 行动 |
|------|------|------|
| 收藏增长加速 | 正向验证：当前方向有效 | 02a保持当前节奏/写法 |
| 收藏增长平稳 | 中性：节奏正常但无爆发点 | 02a在后续5章内设计1个高潮/反转刺激增长 |
| 收藏增长放缓 | 预警：可能进入疲劳期 | 02a调整节奏，引入新冲突/新角色/新场景（INC要素补充） |
| 收藏负增长 | 紧急：读者在流失 | 触发完读率<50%紧急诊断流程 |

#### 3.2 推荐票集中分析

**分析目标**：识别高光章节特征，复制成功模式。

1. 统计推荐票集中的章节（高于均值2倍以上）
2. 分析这些章节的共同特征：
   - 是否为高潮章/反转章/爽点章？
   - 钩子类型是什么？
   - 情绪锚点交付了什么？
   - INC要素命中了哪几项？
3. 将高光章节特征提炼为"高光模板"
4. 注入02a，后续章节按一定频率（每5-10章）安排高光章节

#### 3.3 用户情绪锚点验证（V2.0新增 — 与V5.2机制联动）

> **验证目标**：完读率高的章节是否交付了02a设定的用户情绪锚点？完读率低的章节是否缺失情绪锚点？

**验证流程**：
1. 从02a细纲中提取每章设定的"用户情绪锚点"（V5.1新增字段）
2. 对照完读率数据：
   - 完读率>80%的章节 → 检查是否成功交付了设定的情绪锚点
   - 完读率<50%的章节 → 检查是否缺失情绪锚点或锚点无效
3. 交叉分析：
   - 高完读率 + 情绪锚点交付 → 标记为"有效锚点模式"，注入02a后续章节
   - 高完读率 + 情绪锚点未交付 → 标记为"意外有效"，分析其他原因
   - 低完读率 + 情绪锚点未交付 → 标记为"锚点缺失"，02a后续章节强化情绪工程
   - 低完读率 + 情绪锚点已交付 → 标记为"锚点无效"，分析锚点设置问题
4. 输出情绪锚点验证报告

## 执行率优化说明 V6.0.1

> **优化目标**：将本步骤LLM执行率从70%提升至75%+。
> **优化方式**：补全三级降级链的执行细节——原降级链仅说明"降级至Level 2/3"，但未提供各Level的具体执行方案（Level 1人工输入模板/Level 2 WebSearch关键词/Level 3定性推测方法论），导致LLM在私有数据获取环节卡壳。本次新增4项补全内容。

### 本次优化清单

| 优化点 | 位置 | 优化内容 | 预期效果 |
|--------|------|----------|----------|
| 人工数据输入模板 | Level 1增强（新增） | 提供番茄后台数据填写模板（完读率/追读率/书架比/催更人数/章评/弃书节点）+ 字段说明（标注番茄后台对应页面） | LLM从"不知道该问用户要什么数据"→"按模板引导用户提供结构化数据" |
| WebSearch关键词优化 | Level 2增强（新增） | 提供5类关键词组合表（读者评论/吐槽差评/读后感/社交媒体/数据排行）+ 搜索策略建议 | 提升WebSearch获取公开评论的命中率，从"搜不到"→"按关键词组合精准搜索" |
| 定性推测方法论 | Level 3增强（新增） | 提供7维度推测表（节奏密度/钩子有效性/情绪锚点/信息密度/角色一致性/爽点频率/高光章节）+ 5步执行流程 | LLM从"凭空臆断读者反应"→"按正文特征指标有据推测" |
| 数据质量分级标注 | 新增 | 三级数据质量分级（高=后台数据/中=公开评论/低=定性推测）+ 各级可支撑的调整决策（S1/S2/S3级Findings） | 确保下游02a调整建议的置信度可追溯，避免低置信度数据触发过度调整 |

### 优化原则

1. **不删除原有内容**：原三级降级链（Level 1/2/3）和番茄读者反馈标准化模板（V5.2）保持不变，新增内容作为各Level的执行细节补充
2. **模板与番茄后台对齐**：人工输入模板的字段标注了番茄后台对应页面位置，降低用户填写门槛
3. **推测有据可依**：定性推测方法论基于02a细纲的可量化指标（场景比例/钩子类型/INC命中数），非主观臆断
4. **分级与Findings Schema对齐**：数据质量分级直接映射到Findings Schema的`data_confidence`字段，与09审稿师输出格式一致

### 预期执行率提升路径

- Level 1数据获取执行率：60% → 78%（人工输入模板+字段说明降低用户填写门槛）
- Level 2 WebSearch执行率：55% → 72%（关键词组合表提升搜索命中率）
- Level 3定性推测执行率：50% → 70%（7维度推测表+5步流程使推测有据可依）
- 全步骤综合执行率：70% → 76%（达成目标75%+）
