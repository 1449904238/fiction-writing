# 复杂度冻结协议（V6.0 新增）

> **核心问题**：skill 从 V1.0 到 V5.4 经历了 10+ 个版本迭代，功能不断叠加但从未做过系统性"减法"。文件数量从 15 个膨胀到 50+，规则从 5 条扩展到 25 条，检测器从 3 个增长到 15 个。复杂度的持续增长导致：
> 1. LLM 执行率下降（规则太多记不住/选择性执行）
> 2. 冲突概率上升（新规则与旧规则阈值不一致）
> 3. 维护成本指数增长（改一个阈值需检查 10+ 个文件）
>
> **解决方案**：引入"复杂度冻结"机制——对已稳定的功能标记为"冻结"状态，限制变更频率和范围，确保新功能以"替换"而非"叠加"方式引入。

---

## 一、复杂度分层标记

所有 skill 文件、规则、脚本必须标记复杂度层级：

| 层级 | 标记 | 含义 | 变更限制 |
|------|------|------|---------|
| **FROZEN** | `[FROZEN]` | 已稳定，经过 ≥2 个项目验证 | 仅允许修复 bug，不允许新增功能/调整阈值。年变更 ≤2 次 |
| **ACTIVE** | `[ACTIVE]` | 当前迭代中，正在验证 | 允许功能调整和阈值优化。月变更 ≤4 次 |
| **EXPERIMENTAL** | `[EXPERIMENTAL]` | 新增功能，尚未验证 | 允许自由变更。验证通过后升级为 ACTIVE |

### 文件层级标记规则

在文件头部的 frontmatter 或注释中标注：

```markdown
<!-- 复杂度层级: FROZEN | 最后验证: 2026-07-15 | 验证项目: 3个 -->
```

```json
{
  "complexity_tier": "FROZEN",
  "last_validated": "2026-07-15",
  "validated_projects": 3
}
```

## 二、变更预算

每个版本的变更量受预算约束，防止"大版本炸裂"：

| 版本类型 | 预算 | 允许变更 |
|---------|------|---------|
| **MAJOR**（X.0） | 20个变更点 | 新增文件≤5/修改文件≤10/删除文件≤3/规则变更≤5 |
| **MINOR**（X.Y） | 10个变更点 | 新增文件≤2/修改文件≤5/规则变更≤2 |
| **PATCH**（X.Y.Z） | 3个变更点 | 仅bug修复/文档修正/阈值微调 |

## 三、回归测试框架

### 测试层级

| 层级 | 测试内容 | 执行时机 | 通过标准 |
|------|---------|---------|---------|
| **L0 冒烟测试** | 脚本可执行性（所有 scripts/ 下 .js 文件无语法错误） | 每次提交 | 100%通过 |
| **L1 阈值一致性** | 硬约束总清单 vs rules.json vs prose-utils.js 阈值一致 | 每次规则变更 | 0冲突 |
| **L2 引用完整性** | SKILL.md YAML references 中列出的文件全部存在 | 每次提交 | 100%存在 |
| **L3 场景触发完整** | scene-trigger-map.md 中列出的文件全部存在 | 每次提交 | 100%存在 |
| **L4 版本号一致** | SKILL.md version / rules.json version / 各文件头部版本号一致 | 每次版本发布 | 0冲突 |
| **L5 样本回归** | 使用 samples/ 中的样本章节运行完整流水线，评分不下降 | 每次MAJOR版本 | 评分≥上次90% |

### 自动化检测脚本

```bash
# L0 冒烟测试
node -e "require('./scripts/check-ai-patterns.js')"
node -e "require('./scripts/check-consistency.js')"
node -e "require('./scripts/check-rule-execution.js')"
node -e "require('./scripts/generate-dashboard.js')"
node -e "require('./scripts/estimate-context-load.js')"
node -e "require('./scripts/build-index.js')"

# L1 阈值一致性检查
node scripts/sync-versions.js --check-thresholds

# L2 引用完整性检查
node scripts/sync-versions.js --check-references

# L3 场景触发完整性检查
node scripts/sync-versions.js --check-scene-map

# L4 版本号一致性检查
node scripts/sync-versions.js --check-versions
```

> **L5 样本回归** 为手动测试，使用 `samples/` 目录下的样本章节，运行 03a→03b→04→05 完整流程，对比评分是否下降。

## 四、新功能引入规则

引入新功能时必须遵循"替换优先于叠加"原则：

1. **新增规则前**：检查现有25条规则中是否有覆盖类似场景的规则。如有，修改现有规则而非新增。
2. **新增检测器前**：检查 check-ai-patterns.js 中是否有类似检测。如有，扩展现有检测器而非新建。
3. **新增脚本前**：检查现有脚本是否可通过参数扩展覆盖新需求。如有，添加参数而非新建。
4. **新增references文件前**：检查现有综合文件是否可通过追加章节覆盖。如有，追加而非新建。
5. **新增hooks前**：检查现有hooks是否可通过扩展覆盖。如有，扩展而非新建。

> **强制约束**：每个MAJOR版本新增文件数 ≤5。如超出，必须删除等量的旧文件（合并或归档）。

## 五、冻结解除流程

如需修改 FROZEN 状态的文件/规则：

1. **提案**：在项目日志中记录修改提案（修改内容/原因/影响范围/回退方案）
2. **评审**：用户确认修改必要性
3. **降级**：将文件从 FROZEN 降级为 ACTIVE
4. **修改**：执行修改
5. **验证**：运行 L0-L4 回归测试
6. **升级**：验证通过后恢复为 FROZEN
7. **记录**：在 CHANGELOG.md 中记录本次冻结解除
