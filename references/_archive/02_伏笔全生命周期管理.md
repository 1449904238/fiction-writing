# 伏笔全生命周期管理系统

> **核心定位**：解决百万字长篇中 100-300+ 个伏笔的追踪问题。当前 02 衔接包的"记忆锚点 P0/P1/P2"是状态快照，不是伏笔追踪——没有伏笔 ID、没有生命周期、没有回收检测、没有过期告警。
>
> **痛点对症**：30 章以内靠人脑还能记住伏笔，百万字长篇中"第 800 章忘了第 3 章的伏笔"是必然发生的。业界最佳实践是全生命周期管理：埋设 PLANT → 标记 TAG → 追踪 TRACK → 回收 REAP → 归档。
>
> **理论依据**：每条伏笔分配唯一 ID（如 FS-047），记录埋设章节/角色/类型/预计回收章节/关联伏笔。含轮换间隔（避免连续N章提同一伏笔）、回收标记（防重复回收）、过期告警。

---

## 一、伏笔五态生命周期

```
【伏笔五态生命周期】

State 1: PLANT（埋设）
├─ 触发：02细纲设计时创建伏笔条目
├─ 记录：ID/埋设章/角色/类型/预计回收章/关联伏笔/重要度
└─ 状态：待推进

State 2: ADVANCE（推进）
├─ 触发：后续章节中提及或部分揭示该伏笔
├─ 记录：推进章/推进方式（提及/暗示/部分揭示）
├─ 轮换控制：同一条伏笔不能连续N章提及（避免水字数感）
└─ 状态：推进中

State 3: HINT（暗示）
├─ 触发：给出更明确的线索但不完全揭示
├─ 记录：暗示章/暗示内容
└─ 状态：即将回收

State 4: REAP（回收）
├─ 触发：伏笔真相揭示/兑现
├─ 记录：回收章/回收方式/是否满足读者预期
├─ 防重复：回收后标记为已回收，后续不得再次回收
└─ 状态：已回收

State 5: ARCHIVE（归档）
├─ 触发：回收后经过一定章节数（默认10章）
├─ 记录：归档原因（已回收/已废弃/已合并）
└─ 状态：已归档（不再追踪，但保留记录供查阅）
```

## 二、伏笔追踪表格式

> 02 细纲编写时，每卷维护一张伏笔追踪表。03 扩写时读取本表，扩写后更新推进/回收记录。

### 2.1 伏笔追踪表模板

```
【伏笔追踪表 — 第X卷】

| ID | 伏笔摘要 | 埋设章 | 类型 | 重要度 | 预计回收章 | 当前状态 | 最后提及章 | 轮换间隔 | 关联伏笔 |
|----|---------|-------|------|--------|----------|---------|----------|---------|---------|
| FS-001 | 主角戒指内的老者身份 | Ch.1 | 身份 | P0 | Ch.50 | 推进中 | Ch.12 | ≥5章 | FS-003 |
| FS-002 | 师妹的神秘伤口 | Ch.3 | 悬念 | P1 | Ch.20 | 待推进 | — | ≥8章 | — |
| FS-003 | 戒指老者与师门的关联 | Ch.8 | 关系 | P0 | Ch.50 | 待推进 | — | ≥5章 | FS-001 |
```

### 2.2 字段说明

| 字段 | 说明 | 取值 |
|------|------|------|
| ID | 唯一标识符 | FS-XXX（FS=Foreshadow） |
| 伏笔摘要 | 一句话描述 | 不超过20字 |
| 埋设章 | 首次出现章节 | Ch.X |
| 类型 | 伏笔分类 | 身份/悬念/关系/物品/能力/预言/因果 |
| 重要度 | 对主线的影响 | P0（主线级）/P1（支线级）/P2（细节级） |
| 预计回收章 | 计划回收的章节 | Ch.X 或 Ch.X-Y（区间） |
| 当前状态 | 生命周期状态 | 待推进/推进中/即将回收/已回收/已归档 |
| 最后提及章 | 最近一次被提及的章节 | Ch.X |
| 轮换间隔 | 连续两次提及的最小间隔 | ≥N章 |
| 关联伏笔 | 与其他伏笔的关联 | FS-XXX |

## 三、伏笔检测规则

### 3.1 轮换间隔检测

```
规则：同一条伏笔的连续两次提及之间，间隔不得小于"轮换间隔"值。
违反：标记"伏笔轮换过密"——读者会感觉在水字数。
例外：回收章前1-2章的密集暗示不受此限（蓄势期允许密集提及）。
```

### 3.2 过期告警

```
规则：
- P0级伏笔：超过预计回收章10章未回收 → 告警"P0伏笔过期"
- P1级伏笔：超过预计回收章20章未回收 → 告警"P1伏笔过期"
- P2级伏笔：超过预计回收章30章未回收 → 提示"P2伏笔可考虑废弃"

处理：
- 告警后02细纲需决定：延期回收（更新预计回收章）/ 废弃（标记归档）/ 合并到其他伏笔
- 不得忽略告警——忽略=读者会觉得"挖坑不填"
```

### 3.3 重复回收检测

```
规则：已标记为"已回收"的伏笔，后续章节不得再次回收。
违反：标记"伏笔重复回收"——这会导致逻辑矛盾或读者困惑。
例外：可以"二次揭示"（同一伏笔的更深层次真相），但必须创建新的伏笔ID关联到原伏笔。
```

### 3.4 回收满意度检测

```
规则：伏笔回收时，须检测"是否满足读者预期"。
检测项：
- 回收方式是否与埋设时的暗示方向一致（不一致=反转，需额外铺垫）
- 回收时机是否合理（太早=缺乏期待感，太晚=读者已忘记）
- 回收力度是否匹配重要度（P0伏笔回收需大高潮，P2伏笔一笔带过即可）
```

## 四、02 细纲集成点

### 4.1 02 细纲编写时

每卷细纲须包含本卷的伏笔追踪表：
- 新埋设的伏笔：创建新条目，状态=待推进
- 推进的伏笔：更新最后提及章和状态
- 回收的伏笔：更新状态为已回收，记录回收章和方式
- 过期的伏笔：根据过期告警规则处理

### 4.2 02 衔接包新增字段

在衔接包的"记忆锚点"区域新增"伏笔状态快照"：

```
【衔接包 — 伏笔状态快照】
本卷活跃伏笔数：X
P0级未回收：X（FS-001/FS-003/...）
本章涉及的伏笔：
- FS-001：推进（提及戒指老者的声音）
- FS-002：回收（揭示师妹伤口的来源）
⚠️ 03扩写注意：以上伏笔的状态须在正文中体现，不得遗漏或矛盾。
```

### 4.3 03 扩写的伏笔义务

03 扩写时：
1. 读取衔接包中的伏笔状态快照
2. 按快照在正文中推进/回收对应伏笔
3. 扩写后更新伏笔追踪表（在扩写意图包中记录本章伏笔操作）

## 五、与 check-rhythm.js 的联动

check-rhythm.js 的"篇幅配比"检测会扫描伏笔关键词密度。本系统提供结构化的伏笔追踪，两者互补：
- check-rhythm.js：从文本统计层面检测伏笔密度是否失衡
- 本系统：从设计层面追踪每个伏笔的生命周期

建议在 05 去AI味或 09 审稿时，同时运行 check-rhythm.js 和本系统的伏笔追踪表交叉校验。

## 六、速查卡

```
【伏笔管理速查卡 — 02细纲编写时贴在屏幕上】

每埋一个伏笔，问自己4个问题：
① 这个伏笔什么时候回收？（不填预计回收章=挖坑不填）
② 这个伏笔的轮换间隔是多少？（连续提及=水字数）
③ 这个伏笔和哪些伏笔关联？（孤立的伏笔=浪费篇幅）
④ 这个伏笔的回收力度够不够？（P0伏笔回收需大高潮）

每回收一个伏笔，问自己3个问题：
① 回收方式是否与暗示方向一致？（不一致=需额外铺垫反转）
② 这个伏笔之前已经回收过了吗？（重复回收=逻辑矛盾）
③ 回收后读者会满足还是失望？（失望=回收力度不够或时机不对）
```
