# 03→04→05 交接协议

> **核心定位**：解决三步分离导致的上下文断层。02→03 有衔接包做数据传递，但 03→04、04→05 之间此前没有任何交接协议——03 扩写时设计的情绪通路/锚点/口癖意图到 04 全部丢失，04 精修时打磨的金句位置/潜台词/风格指纹到 05 又丢失。
>
> **本协议定义两种交接包**：
> - **扩写意图包**（03→04）：03 扩写完成后随正文传递给 04，记录 03 的设计意图
> - **精修意图包**（04→05）：04 精修完成后随正文传递给 05，记录 04 的打磨决策
>
> **核心原则**：交接包是"设计意图的载体"，不是"施工图纸"。下游步骤读交接包是为了"理解上游为什么这么写"，从而避免推翻上游的有意设计。

---

## 一、扩写意图包（03→04 交接）

> 03 扩写完成后，在正文末尾以结构化注释形式输出扩写意图包，04 精修时必须先读取本包再开始精修。

### 1.2 04 精修师的读取义务

04 精修师在开始精修前，必须：
1. **先读取扩写意图包**，理解 03 的情绪设计、角色口癖、刻意设计项
2. **标记"禁止删除区"**：扩写意图包中的"刻意设计项"和"情绪锚点"不得当 AI 味删除
3. **聚焦"已知不足"**：03 标记的不足项是 04 的工作重点
4. **精修完成后删除意图包注释**：意图包是内部交接用的，不保留在最终正文中

## 二、精修意图包（04→05 交接）

> 04 精修完成后，在正文末尾以结构化注释形式输出精修意图包，05 去AI味时必须先读取本包再开始检测。


### 2.2 05 去AI味的读取义务

05 去AI味在开始检测前，必须：
1. **先读取精修意图包**，理解 04 的金句位置、潜台词设计、风格指纹、刻意保留项
2. **跳过"已处理项"的全量重扫**：04 标记已处理的项，05 只做终检抽样，不做全量重扫
3. **保护"刻意保留的'不完美'"**：04 标记的刻意保留项不得当 AI 味修正
4. **聚焦"已知残留风险"**：04 标记的风险项是 05 的工作重点
5. **去AI味完成后删除意图包注释**：意图包不保留在最终正文中

### 2.3 精修意图包结构化字段定义（V5.3新增）

> 上述 2.1 模板为自然语言注释版，适用于 AI 直接阅读。本节定义结构化 JSON 版本，适用于确定性脚本校验和多 Agent 系统间数据传递。Schema 文件见 `schemas/refinement-intent.schema.json`。

#### 2.3.1 字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `chapter_no` | integer | 是 | 章节号 |
| `processed_items` | array | 是 | 04 已处理项清单，05 可据此跳过初检 |
| `processed_items[].category` | enum | 是 | 处理类别：`psychological_direct`(心理直给) / `explanation_tone`(解释腔) / `metaphor_quality`(意象质量) / `sensory_redirect`(感官重定向) / `dialogue_subtext`(对话潜台词) |
| `processed_items[].status` | enum | 是 | 处理状态：`fully_processed`(已完全处理，05可跳过初检) / `partially_processed`(部分处理，05需抽样复核) / `skipped`(跳过未处理，05必须全量检测) |
| `processed_items[].gate_id` | string | 是 | 对应的 05 Gate 编号（如 "Gate 3"、"Gate 11"） |
| `processed_items[].note` | string | 否 | 处理说明（如"已将第3段心理直给外化为手部动作"） |
| `boundary_ambiguous_items` | array | 否 | 04 标记的边界模糊项——04 判断为有意设计但可能触发 05 Gate 的位置，05 仅检测这些位置而非全量扫描 |
| `boundary_ambiguous_items[].location` | string | 是 | 位置标注（如"第5段第2句"、"章末排比句"） |
| `boundary_ambiguous_items[].description` | string | 是 | 边界模糊说明（如"此处排比是情绪爆发设计，非AI升华"） |
| `boundary_ambiguous_items[].gate_id` | string | 是 | 对应的 05 Gate 编号 |

#### 2.3.2 与 05 Gate 的映射关系

| category | 对应 05 Gate | 跳过初检条件 | 边界模糊检测逻辑 |
|----------|------------|------------|----------------|
| `psychological_direct` | Gate 3 | status=fully_processed → 跳过初检 | boundary_ambiguous_items 中 gate_id="Gate 3" 的位置 → 仅检测这些位置 |
| `explanation_tone` | Gate 11 | status=fully_processed → 跳过初检 | boundary_ambiguous_items 中 gate_id="Gate 11" 的位置 → 仅检测这些位置 |
| `metaphor_quality` | Gate 7 | status=fully_processed → 跳过初检 | boundary_ambiguous_items 中 gate_id="Gate 7" 的位置 → 仅检测这些位置 |
| `sensory_redirect` | Gate 3（感官部分） | status=fully_processed → 跳过初检 | — |
| `dialogue_subtext` | Gate 9 | status=fully_processed → 跳过初检 | boundary_ambiguous_items 中 gate_id="Gate 9" 的位置 → 仅检测这些位置 |

#### 2.3.3 降级策略

- **有意图包**：05 按 processed_items 的 status 决定跳过/抽样/全量，按 boundary_ambiguous_items 仅检测标记位置
- **无意图包**：05 降级为全量 11 Gate 检测（参见 3.3 异常处理）
- **意图包格式损坏**：按无意图包处理，记录日志

## 三、交接包的传递机制

### 3.1 传递方式

交接包以 HTML 注释形式嵌入正文末尾（`<!-- ... -->`），原因：
- 不影响正文阅读
- 可被 AI 识别为结构化数据
- 落盘后可被确定性脚本检测（check-degeneration.js 已豁免注释块）

### 3.2 生命周期

```
03扩写 → 输出正文 + 扩写意图包
         ↓
04精修 → 读取扩写意图包 → 精修正文 → 删除扩写意图包 → 输出正文 + 精修意图包
         ↓
05去AI味 → 读取精修意图包 → 检测修正 → 删除精修意图包 → 输出最终正文
```

### 3.3 异常处理

| 异常情况 | 处理方式 |
|---------|---------|
| 04 收到正文但无扩写意图包 | 04 从文本反推 03 意图，在精修意图包中标注"03意图包缺失，以下为04推断" |
| 05 收到正文但无精修意图包 | 05 执行全量 11 Gate 检测（不做跳过优化），标注"04意图包缺失，全量重扫" |
| 意图包格式损坏 | 按无意图包处理，记录日志 |

## 四、与现有机制的关系

| 现有机制 | 定位 | 与本协议关系 |
|---------|------|------------|
| 02 衔接包 | 02→03 的数据传递 | 本协议是 03→04→05 的数据传递，与衔接包形成完整的"三段交接链" |
| 03 四维锚点确认块 | 03 扩写前的设计确认 | 四维锚点是"设计阶段"，扩写意图包是"执行后的交接"——前者是计划，后者是交付 |
| 05 分级执行策略 | 05 的 Gate 范围选择 | 精修意图包的"已处理项"为分级执行提供额外信息——已处理项可降低检测优先级 |
| 05 白名单豁免 | 05 的术语保护 | 精修意图包的"刻意保留项"是动态白名单——比静态白名单更精准 |
